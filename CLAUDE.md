# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

A Bun monorepo managed by **Turborepo** containing the blog app, a CLI, and the shared packages they depend on, by Howard Tai.

- `apps/`: `blog`, `cli`
- `packages/`: `ui`, `test-config`, `tsconfig`, `article-contract`

## Commands

### Root (run all packages)

```bash
bun run build       # build all packages (respects dependency order via turborepo)
bun run lint        # lint all packages
bun run test        # run all tests
bun run type-check  # TypeScript check across all packages
bun run format      # ultracite fix
```

### Per-package (cd into app or package first)

```bash
bun run dev         # start dev server
bun run build       # build
bun run lint        # lint
bun run test        # run bun tests
bun run type-check  # tsc --noEmit
```

Run a single test file:
```bash
bun test <filename>
```

### Blog app extras (`apps/blog`)

```bash
bun run analyze           # production build with @next/bundle-analyzer
```

## Architecture

### Monorepo tooling

- **Bun workspaces** for dependency management
- **Turborepo** (`turbo.json`) as the task runner ā€” build order is dependency-aware, caches `lint`, `type-check`, `test`, `build`
- **Git worktrees**: after creating a worktree of this repo, run `bun install` inside it before running any workspace script ā€” workspace module resolution is broken otherwise.

### Shared packages

| Package | Purpose |
|---|---|
| `@howardism/ui` | shadcn/ui components (Tailwind v4) |
| `@howardism/test-config` | Shared Bun test preload (happy-dom, jest-dom matchers, Next.js mocks) |
| `@howardism/tsconfig` | Shared TypeScript configs |
| `@howardism/article-contract` | Zod schemas, `WikiDomain`/`WikiTag` unions, and manifest types for the wiki article contract ā€” the source of truth shared by the CLI (write side, emits manifests) and the blog (read side, validates on read) |

### Blog app (`apps/blog`)

**Next.js 16** (App Router) + **React 19** + **Tailwind v4**. Articles-focused: no auth, no database, no API routes. Routes: home, `/articles`, `/articles/[slug]`, `/articles/domain/[domain]`, `/articles/tag/[tag]`, `/articles/tagged/[tag]`, `/compare` (side-by-side article view), `/shelf` (browser-local reading history), `/questions` (open-questions worklist), RSS feeds ā€” plus a `zh-TW` locale mirror at `/[locale]/articles` and `/[locale]/articles/[slug]`. (`/photos`, `/about`, and `/thank-you` were removed; `next.config.ts` permanently redirects all three to `/`.)

Key internal structure under `src/`:
- `app/(blog)/` ā€” pages and the `(layout)` group (Header, Footer)
- `app/(common)/` ā€” shared layout/UI primitives (`Container`, `SimpleLayout`, `Card`, icons, ā€¦)
- `app/rss/` ā€” RSS feed routes (`feed.xml`, `feed.json`)
- `config/` ā€” env validation via a raw zod schema (`env.ts`) and `security-headers.ts` (CSP, consumed by `next.config.ts`)
- `components/`, `hooks/`, `utils/`, `types/` ā€” UI and shared helpers

Articles: MDX, glob-discovered from local files (`articles/service.ts`), statically generated. Each carries a `tag` (article *kind* ā€” Concept/Entity/Essay/Index, drives `/articles/tag/[tag]`), a `domain` (curated knowledge domain, drives `/articles/domain/[domain]`), and optional free-form `tags` (subject labels, drive `/articles/tagged/[tag]`). Most articles ā€” and the `src/data/*.json` manifests the blog reads at build time ā€” are generated by the `apps/cli` wiki importer (see below).

Design system: a single set of oklch design tokens in shadcn slots (`--background`, `--foreground`, `--primary`, ā€¦) plus `--brand*` for the editorial terracotta accent, defined in `@howardism/ui`'s `globals.css`; light/dark only, toggled via a Tweaks panel (persisted in `localStorage`). Article bodies use the `@tailwindcss/typography` `prose` plugin, themed onto those tokens. `apps/blog/src/styles/howardism.css` holds only the paper-grain body background and a couple of CSS-only effects.

### CLI (`apps/cli`)

`bun run import:wiki` (entry `src/import-wiki/index.ts`) parses an Obsidian-style wiki vault ā€” point `WIKI_PATH` (and optionally `RAW_PATH`) at it ā€” and emits article MDX/frontmatter into the blog plus three committed manifests under `apps/blog/src/data/`: `article-graph.json` (backlink/related graph), `wiki-sources.json` (per-article external source audit trail), and `open-questions.json` (the vault's open-questions backlog, regrouped by domain for the `/questions` page). Two more manifests in the same directory come from separate commands: `search-index.json`, built from the committed MDX articles by `bun run build:search-index`, and `translations.json`, the current-state translation record per slug maintained by `bun run translate`. Domains are resolved from the vault's `moc-*` Map-of-Content membership (derived notes fall back to a catch-all domain). The blog reads all five JSON files at build time, so re-run the relevant command and commit the result when source notes or articles change. For a full refresh (import + hero image generation + validation), use the `generate-wiki-images` skill in `.claude/skills/`.

#### Content integrity gate

`bun run content:check` ā€” also run in CI ā€” fails on missing hero images (MDXā†”PNG mismatch), broken slug references in `article-graph.json`/manifests, and missing `title`/`description`/`imageAlt` frontmatter. It warns (non-fatal) on orphan articles (no backlinks), domains without a `moc-<domain>` article, and orphan PNGs.

#### Translation glossary (SQLite + MCP)

The translation service (`bun run translate`) keeps a **do-not-translate (DNT) glossary** of proper nouns and technical terms to leave verbatim. It lives in `src/glossary/` backed by SQLite (`bun:sqlite`, WAL) at `apps/cli/.translate-glossary.db` ā€” a per-machine, gitignored cache. `store.ts` is the shared core (`openDb`, `listTerms`, `addTerm`, `searchTerms`, `ensureSeeded`); `seed.ts` harvests seed terms (Entity-tagged article titles+slugs, wiki-source authors, base tech acronyms). On first use `ensureSeeded` migrates any legacy `.translate-glossary.json` then harvests the corpus; a populated DB is left untouched so added terms persist.

`bun run translate:check` reports translation staleness across six buckets ā€” **Fresh**, **Stale**, **Verbatim-drift**, **Missing**, **Orphan**, **Untranslated** ā€” by comparing each translated article against its English source; CI runs it with `--warn` to post annotations instead of failing the build. `bun run translate:drip --limit N --interval-min M --max-cycles K` is the fix: a quota-paced loop that runs `translate --update` in small batches spaced across engine quota windows (agy/codex exhaust quota mid-run), repeating until `translate:check` reports nothing actionable; it runs `glossary harvest --add` once at startup.

Two interfaces share the one DB:
- **CLI** ā€” `bun run glossary list` (JSON array of `{term, category}`), `bun run glossary add "<term>" <category>`, and `bun run glossary harvest [--add] [slug ...]` (pre-scans article MDX for DNT candidates ā€” acronyms, proper nouns ā€” and diffs against the DB; dry-run by default, `--add` registers). The translation prompt instructs each engine to shell out to these.
- **MCP server** ā€” `bun run glossary:mcp` (stdio) exposes `glossary_list`, `glossary_add`, `glossary_search` so any MCP client can reuse the glossary as native tool calls. Registered for this repo in `.mcp.json`; other agents can point at it with:

  ```json
  { "mcpServers": { "glossary": { "command": "bun", "args": ["apps/cli/src/glossary/mcp.ts"] } } }
  ```
