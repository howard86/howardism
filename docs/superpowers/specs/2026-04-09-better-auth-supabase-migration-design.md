# Migrate next-auth to Better Auth with Supabase PostgreSQL

## Summary

Clean-slate migration of the blog app's authentication from next-auth v4 (JWT sessions, SQLite) to Better Auth (database-backed sessions, PostgreSQL via Supabase). Includes replacing the SQLite database with Supabase PostgreSQL for the entire Prisma schema.

## Motivation

- next-auth v4 is aging; Better Auth offers a more modern, type-safe API
- Database-backed sessions enable session revocation and device tracking
- SQLite is not suitable for production deployment; Supabase PostgreSQL provides a managed, scalable database
- The auth surface is small (6 files), making this a low-risk migration window

## Approach

**Clean-Slate Migration** ‚Äî delete all next-auth code and SQLite config, build Better Auth + Supabase PostgreSQL from scratch. No incremental rollout or parallel setup needed given the small auth surface.

## Architecture

### Before ‚Üí After

| Component | Before | After |
|---|---|---|
| Auth library | `next-auth` v4 | `better-auth` |
| Database | SQLite via `better-sqlite3` | PostgreSQL via Supabase |
| Prisma adapter | `@prisma/adapter-better-sqlite3` | `better-auth/adapters/prisma` |
| Session strategy | JWT (stateless) | Database-backed sessions (cookies) |
| Auth API route | Pages Router `src/pages/api/auth/[...nextauth].ts` | App Router `src/app/api/auth/[...all]/route.ts` |
| Preview auth | Hardcoded Credentials provider | `emailAndPassword` plugin with seeded test user |
| Route protection | `next-auth/middleware` in `proxy.ts` | Per-page server checks via `auth.api.getSession()` |
| Prisma client | Custom SQLite adapter in `src/services/prisma.ts` | Standard `PrismaClient` with `DATABASE_URL` + `DIRECT_URL` |
| Login UI | next-auth default sign-in page | Custom minimal login page |

## Database & Prisma

### Datasource

```prisma
datasource db {
  provider  = "postgresql"
  url       = env("DATABASE_URL")
  directUrl = env("DIRECT_URL")
}
```

### Auth Schema Changes

Better Auth requires these tables, replacing the current next-auth ones:

**`user`** ‚Äî `emailVerified` changes from `DateTime?` to `Boolean`; adds `createdAt`, `updatedAt`; `name` and `email` become required.

**`session`** ‚Äî `sessionToken` ‚Üí `token`; `expires` ‚Üí `expiresAt`; adds `ipAddress`, `userAgent`, `createdAt`, `updatedAt`.

**`account`** ‚Äî snake_case ‚Üí camelCase fields; `provider` ‚Üí `providerId`; adds `accountId`, `password`, `accessTokenExpiresAt`, `refreshTokenExpiresAt`, `createdAt`, `updatedAt`; removes `type`, `token_type`, `session_state`.

**`verification`** ‚Äî replaces `VerificationToken`; single `id` PK; `token` ‚Üí `value`; `expires` ‚Üí `expiresAt`; adds `createdAt`, `updatedAt`.

The exact schema will be generated by Better Auth's CLI (`bunx auth generate`) and merged into the existing Prisma schema file.

### Business Models

Resume and Commerce models stay structurally unchanged. The `Json` fields (`responsibilities`, `subjects`, `descriptions`, `items`) map naturally to PostgreSQL's `jsonb`.

### Prisma Client

`src/services/prisma.ts` simplifies to a standard `PrismaClient()` ‚Äî no more `PrismaBetterSqlite3` adapter, `getDatabaseUrl()` parsing, or `file:` prefix stripping.

### Supabase Setup

1. Create a dedicated `prisma` database user via SQL Editor with `bypassrls`, `createdb`, and grants on `public` schema
2. Two connection strings in `.env`:
   - `DATABASE_URL` ‚Äî pooled (port 6543, `?pgbouncer=true`) for application queries
   - `DIRECT_URL` ‚Äî session mode (port 5432) for migrations

### Migration Strategy

Fresh database ‚Äî use `bunx prisma migrate dev` to create all tables from scratch. No data migration needed; existing SQLite data is dev/seed data only.

## Auth Server & Client

### Server Instance (`src/lib/auth.ts`)

```typescript
import { betterAuth } from "better-auth";
import { prismaAdapter } from "better-auth/adapters/prisma";
import prisma from "@/services/prisma";
import { env } from "@/config/env.mjs";

export const auth = betterAuth({
  database: prismaAdapter(prisma, {
    provider: "postgresql",
  }),
  emailAndPassword: {
    enabled: true,
  },
  socialProviders: {
    google: {
      clientId: env.GOOGLE_CLIENT_ID,
      clientSecret: env.GOOGLE_CLIENT_SECRET,
    },
    github: {
      clientId: env.GITHUB_ID,
      clientSecret: env.GITHUB_SECRET,
    },
  },
});
```

All providers are always registered ‚Äî no environment-conditional provider array. The login UI can choose which to show per environment if needed.

### Client Instance (`src/lib/auth-client.ts`)

```typescript
import { createAuthClient } from "better-auth/react";

export const authClient = createAuthClient();
```

No `baseURL` needed since client and server share the same domain.

### API Route Handler (`src/app/api/auth/[...all]/route.ts`)

```typescript
import { auth } from "@/lib/auth";
import { toNextJsHandler } from "better-auth/next-js";

export const { POST, GET } = toNextJsHandler(auth);
```

## Callsite Migration

### Server-side session checks

Both `profile/page.tsx` and `profile/resume/[profileId]/utils.tsx` change from:

```typescript
import { getServerSession } from "next-auth/next";
import { authOptions } from "@/pages/api/auth/[...nextauth]";
const session = await getServerSession(authOptions);
```

To:

```typescript
import { headers } from "next/headers";
import { auth } from "@/lib/auth";
const session = await auth.api.getSession({ headers: await headers() });
```

The `session.user` shape is compatible ‚Äî Better Auth's user object has `name`, `email`, `image` by default, so template code stays the same.

### Client-side logout

`LogoutButton.tsx` changes from `signOut()` via `next-auth/react` to `authClient.signOut()` via the Better Auth client.

### Route protection

`src/proxy.ts` (using `next-auth/middleware`) is deleted. Per-page server checks in the profile pages are sufficient. Better Auth explicitly recommends per-page checks over middleware-based session management.

## Login Page

A new minimal login page at `src/app/(blog)/login/page.tsx`:
- Google and GitHub OAuth buttons
- Email/password form
- Redirects to `/profile` on success

Unauthenticated users hitting protected pages (`/profile`, `/profile/resume/*`) are redirected to `/login` instead of `/` (current behavior). This makes the auth flow more intuitive ‚Äî users land on the login page rather than the homepage.

Uses the Better Auth React client:

```typescript
authClient.signIn.social({ provider: "github" });
authClient.signIn.social({ provider: "google" });
authClient.signIn.email({ email, password });
```
