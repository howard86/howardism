## 摘要

[Fiona Fung](/articles/fiona-fung) 在運營 [Claude Code](/articles/claude-code) + [Cowork](/articles/cowork) 工程團隊時提出的核心主張：多年來，**工程產能一直是昂貴的資源**——規劃、審查和流程都是為了保護它而存在。一旦 agentic coding 讓寫程式變得廉價，瓶頸就轉移到了**驗證、審查和維護**。「在 Claude Code 團隊中，寫程式真的不再是慢的部分了。」新的稀缺資源是*對變更正確性的信心*——而這種信心恰恰因為產能（以及吞吐量）爆發而變得更加稀缺。

## 為什麼驗證現在成為制約

三股力量匯聚：

- **數量。** 產能大幅增加，以至於「我們必須更加關注：它是否正確。」
- **角色模糊。** 更多人（設計師、管理者、PM）現在都在提交變更，所以每個人都需要對自己的變更正確性有信心。
- **維護成本。** 更高的吞吐量意味著更多需要維護的東西——維護成本成為一等考量，而非事後才想到的問題。

這是 [Karpathy](/articles/andrej-karpathy) 的 [The Verifiability Thesis](/articles/verifiability-thesis)（「LLM 能自動化你能驗證的事物」）在組織層面的映射，也是 [Harness Shrinkage as Models Improve](/articles/harness-shrinkage-as-models-improve) 的需求面（prompt 腳手架縮減；*機械式驗證仍然是承重結構*）。

## TDD 不再是負擔

這個轉變的一個生動跡象：TDD 過去感覺像「吃花椰菜」——先寫失敗的測試，驗證它確實失敗，然後修復。有了 Claude，Fung 發現它「變得有趣多了、愉快多了……它把測試驅動開發的負擔去除了。」經濟學翻轉了：當寫測試幾乎免費時，奠定驗證基礎的紀律（一個可證明會失敗、然後通過的測試）就是純粹的收益。（參見 `tdd` / red-green-refactor 紀律；先寫失敗測試的步驟就是驗證器。）

## Shift left

她反覆提到的短語：**shift left**——透過自動化在源頭附近捕捉問題，而不是等客戶遇到之後才處理。「什麼比我先遇到 bug 更好？有自動化機制在更接近源頭的地方捕捉它。」隨著吞吐量上升，驗證能跟上的唯一方式是*自動化且提前*，而非手動且滯後。

## 誰來審查——以及人類介入的界線

在推出 Claude Code 自己的程式碼審查功能之前，「你怎麼跟上程式碼審查？」是她被問最多的問題。答案：**Claude Code review** 處理風格、lint、明顯的 bug，以及 *spec 偏移*（如果你把 spec 簽入程式碼庫，「Claude 非常擅長驗證 spec 偏移」）。但人類在重要的地方保持介入：**法律審查、風險容忍度、信任邊界**——「信任但驗證，以及人類帶來所需專業知識的地方。」分工方式：自動化機械式驗證，將人類判斷保留給風險和信任邊界的決策。（參見 [Deep Modules for Agents](/articles/deep-modules-for-agents)：在全新上下文中的審查者。）

## 衡量這個轉變（以及一個陷阱）

她關注的信號：新人上手時間 ↓、PR 週期時間 ↓、Claude 輔助的 commit ↑（「我已經好幾個月沒看到不是 Claude 輔助的 commit 了」）。**陷阱**：不要只看端到端的 PR 週期時間——*把它拆分成漏斗的各個階段*。如果週期時間沒有下降，可能不是 AI 採用率低；可能是 **CI/build 系統在新的吞吐量下卡住了。** 而且吞吐量不是目標——「找到某種方式來衡量你實際想解決的問題」，而不只是速度。

## 開放問題

- Fung 自己的開放問題：「你把全自動審查推到多遠？」——速度與安全的平衡點在哪裡，以及如何讓人類保持信心而不重新引入審查瓶頸？
- 如果 CI/build 是隱藏的堵塞點，驗證基礎設施（測試執行器、CI 容量）是否會成為 AI-native 組織的實際資本支出？
