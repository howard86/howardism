- [How I AI: Thariq Shihipar on Replacing Markdown with HTML for AI-Powered Development](https://www.chatprd.ai/how-i-ai/claude-code-anthropic-thariq-shihipar-on-replacing-markdown-with-html)

## 摘要

[Thariq Shihipar](/articles/thariq-shihipar) 在 Code with Claude（2026 年 5 月）向 [Claire Vo](/articles/claire-vo) 發表的論點：**「HTML 是新的 markdown。」** markdown 一直是人類與 LLM 互動的通用語言——簡單、結構化、雙方都能閱讀。但隨著模型越來越強大、專案越來越複雜，數千行的 markdown 計畫反而壓垮了*人類*。我們開始走馬看花、不再閱讀、不再參與，協作品質隨之崩塌。解決方案是更豐富的溝通媒介：視覺化、可捲動、可互動的 HTML 產物，將人類從被動的審閱者拉回主動的協作者。

關鍵觀察：**模型不是瓶頸——人類的注意力才是。**

> 「不是模型處理不了 Markdown——它們非常擅長——而是 HTML 讓*我們*這些迴圈中的人類更容易視覺化、互動，並與正在進行的工作保持連結。」

## 它修正的失敗模式

Thariq 自己的坦白精準地點出了問題：面對一份數千行的計畫，「他會直接叫 Claude 編輯計畫，而不是自己去讀。」一份人類不再閱讀的計畫，就是一份人類不再引導的計畫——計畫存在的目的是產生對齊，而這個對齊就此蒸發。目標變成「創造一份你真正*想讀*的計畫。」

這是人類注意力版本的模型 [Context Window Smart Zone](/articles/context-window-smart-zone)：就像模型在超過約 100K token 後會退化（無論宣稱的視窗有多大），人類審閱者在面對超過某個量的無差別 markdown 後也會退化（無論內容多完整）。HTML 透過將 token 花在可讀性上（模型圖、結構、色彩、互動）而非純文字，提升了人類的有效 smart zone。

## HTML 帶來什麼

當被要求「用 HTML 檔案腦力激盪一些想法」時，Claude 回傳的不是一份清單，而是一個視覺化、可捲動的頁面——八個示範想法，每個都有模型圖、描述和風險評估。完整計畫的提示詞：

```
create a HTML file as a plan that helps me visualize what the implementation plan is.
Include excerpts, mockups, code, whatever is needed to give me like maximum context
```

產出一個單檔網站：腳本化的介紹、檔案結構圖、程式碼片段、視覺情緒板。一份你可以閱讀、探索，並**分享給隊友**的文件——他們參與這份文件的可能性遠高於一個巨大的 markdown 檔案。這個產物不僅橋接了人類↔AI，也橋接了工程↔公司其他部門。

## 提示技巧：約束加上自由度

計畫提示詞中有兩個細節是刻意的：

- 他拼錯了「excerpts」，但這不影響——模型對表面噪音具有魯棒性。
- **「whatever is needed to give me like maximum context」** 給了模型一個出口。哲學是：提供足夠的約束來得到你想要的，但留下空間讓模型給你驚喜。如 Thariq 所說，*「我總是需要說，『嘿，Claude，我信任你。』」*

這是 [The Bitter Lesson](/articles/the-bitter-lesson) 在提示層級的一個小型、可重複的實例——少指定結構，讓規模化的能力來填補——它與 [Compute Allocator](/articles/compute-allocator) 角色配對：人類設定方向，信任模型處理其餘部分。

## 它解決的 harness 張力

乍看之下，這與本 wiki 持續討論的 [Harness Shrinkage as Models Improve](/articles/harness-shrinkage-as-models-improve) 論點（[Cat Wu](/articles/cat-wu) / [Boris Cherny](/articles/boris-cherny)：腳手架每次發布都在縮小）和 [Interaction Models](/articles/interaction-models)（[Thinking Machines Lab](/articles/thinking-machines-lab)：harness 溶解*進入*模型）相矛盾。Thariq 說的是相反的——他只有約 1% 的 token 會上線；99% 都花在腳手架上。

解答：這是**兩種不同的 harness。**

- **縮小的** harness 是*面向模型的*——提示、指令、驗證，存在的目的是讓模型表現正確。隨著模型進步，這部分收縮。
- **擴大的** harness 是*面向人類的*——產物（HTML 計畫、[micro-apps](/articles/disposable-micro-apps)、[設計系統](/articles/living-design-system)），存在的目的是讓*人類*保持對齊、參與，並能做出決策。隨著模型進步，這部分*擴張*，因為約束瓶頸從「模型能不能做到」轉移到「人類能不能留在迴圈中並妥善分配算力」。

[The Bitter Lesson](/articles/the-bitter-lesson) 溶解面向模型的結構；它不會溶解面向人類的結構——後者的全部目的就是人類理解。這就是 HTML 產物所在的那一側。參見 [Compute Allocator](/articles/compute-allocator) 了解這暗示的角色。

## 三種工作流程

1. **腦力激盪與規劃**以互動式 HTML 呈現（本頁）——你想讀的視覺化計畫。
2. **[Disposable Micro-Apps](/articles/disposable-micro-apps)**——用完即棄的自訂 UI 來*編輯*計畫的部分，回應「HTML 不可編輯」的反對意見。
3. **[Living Design System](/articles/living-design-system)**——一個 `design_system.html` 作為可攜式、人類和機器都能讀取的真實來源。

三者都以人類為中心；沒有一個是為了把開發者自動化掉。

## 開放問題

- 面向人類的 harness 是否會無限制地持續擴大，還是會碰到自己的膨脹天花板（一份太精緻而無法閱讀的 HTML 計畫，就像它所取代的 markdown 一樣）？**已回答：**[Does the Human-Facing Harness (HTML Artifacts) Hit Its Own Bloat Ceiling?](/articles/human-facing-harness-bloat-ceiling) — 是的；HTML 提升並重塑了人類注意力的天花板，但無法移除它，膨脹從文件長度轉移到產物蔓延/橡皮圖章式審批。
- HTML 比 markdown 更難做差異比對和版本控制——當產物是單檔網站時，計畫歷史和審閱會怎樣？（[Disposable Micro-Apps](/articles/disposable-micro-apps) 的回寫至 markdown 是一種補丁。）
- 這是否能推廣到一位專家實踐者之外，還是需要 Thariq 級別的 Claude 流暢度才值得這些額外開銷？
