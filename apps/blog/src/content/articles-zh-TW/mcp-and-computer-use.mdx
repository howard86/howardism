## 摘要

兩種互補的機制，用於將模型連接到外部軟體，皆由 Anthropic 打造，皆承載於 [Claude Code](/articles/claude-code) / [Cowork](/articles/cowork) / Chat 產品介面之上。**MCP**（Model Context Protocol）是結構化的程式存取——「你在 Claude AI 中使用的同一個連接器」可以接入 Salesforce、Google Docs、Google Calendar、Slack、Figma、Gmail，以及越來越多的利基產業系統。**Computer use** 則是針對未提供 MCP 的軟體的**通用方案**：模型直接驅動 GUI（滑鼠、鍵盤、螢幕），速度較慢但在 Opus 4.7 上越來越有能力。Boris Cherny 的框架：*「對模型而言，一切都只是 token」*——MCP / API / computer use 是同一能力的可互換基底。

## MCP 是什麼

由 Anthropic Labs（2024 年底）與 [Claude Code](/articles/claude-code) 及桌面應用程式一同創建，出自 [Boris](/articles/boris-cherny) 的創始團隊。結構化的工具呼叫協定，採用伺服器-客戶端架構：

- **伺服器** ——運行在外部系統旁（Salesforce、Slack、Gmail、內部 CRM、利基產業 SaaS）；將可用工具以型別化函式呼叫的形式暴露出來。
- **客戶端** ——消費這些工具的 Claude 介面（Claude Code、Cowork、Claude AI、第三方 agent）。
- **所有介面共用同一組連接器。**「你在 Claude AI 中使用的同一個 MCP 連接器，你接上 Salesforce，接上 Google Docs、Google Calendar。然後 Cowork 可以用它。Claude CLI 可以用它。Claude Code 到處都能用它。」——Boris Cherny

結構性特質：連接器邏輯只需為每個系統撰寫一次，即可被所有 Claude 介面消費。這正是讓 [Cowork](/articles/cowork) 能夠在既有的知識工作工具表面（Salesforce、Docs、Drive、Slack 等）上運作的關鍵，而無需 Anthropic 為每個工具建構個別整合。

## Computer use 是什麼

當 MCP 不可用時，以通用 GUI 驅動作為備援。模型看到螢幕截圖，決定要點擊/輸入/捲動什麼，透過無障礙/自動化 API 執行。可操作「你電腦上幾乎任何一款軟體」（Boris Cherny）。

截至 Opus 4.7 的特性：
- **品質** ——「相當好……現在做得相當好，尤其是 4.7」（Boris）。[Anthropic](/articles/anthropic)「在電腦操作方面遙遙領先。」
- **延遲** ——「非常慢。」同一任務比 MCP 消耗更多 token，因為每個動作都需要一次螢幕截圖往返。
- **覆蓋範圍** ——通用。當目標軟體沒有 API、沒有 MCP、沒有 Python 函式庫——唯一的介面是面向人類的 UI 時，computer use 就是執行方案。

Cowork 是目前 computer use 最重要的部署介面：許多知識工作應用程式缺乏程式化介面。

## 「無所謂」論點

Boris 對 MCP vs. API vs. computer use 問題的框架：

> 「這些東西其實都不太重要。可以是 MCP、API，或只是某種程式化存取，因為模型不在乎。對模型而言，一切都只是 token。」

基底是可替換的——工作是「以模型能消費的形式向模型暴露能力」。MCP 針對結構化/快速/低成本進行最佳化；computer use 針對通用/備援/慢速進行最佳化。兩者都歸結為 token 層級的工具呼叫。

這連結到 [The Bitter Lesson](/articles/the-bitter-lesson)：隨著模型改進，「使用 MCP」與「使用 computer use」之間的邊界應該是*模型*做出的決定，而非人類 harness 設計者做出的決定。Boris 對未來幾年的預測：

- 「模型將會完成所有程式碼。它將啟動 agent。它將建構環境。」——推測也包括選擇正確的基底來呼叫工具。
- Computer use 被特別點名為一個「將會大幅改善」的產品領域。

## 跨介面實際使用情況

| 介面 | MCP 範例 | Computer use 範例 |
|---|---|---|
| [Claude Code](/articles/claude-code)（CLI） | GitHub、檔案系統、Slack | 罕見——工程工具通常有 CLI/API |
| [Cowork](/articles/cowork) | Salesforce、Google Drive/Docs/Calendar、Gmail、Slack、Figma | 沒有 MCP 的軟體；尤其是知識工作應用程式 |
| Claude AI（聊天） | 同一組連接器 | 可使用 computer use |
| 行動裝置/網頁 | 同一 MCP 基礎設施 | 瀏覽器端，需螢幕分享權限 |

[Cat Wu](/articles/cat-wu) 的每晚簡報製作工作流程（[Cowork](/articles/cowork)）明確使用 MCP——Figma MCP、Slack MCP、Drive MCP——而非 computer use，因為延遲成本對於一個你希望在早上前完成的工作流程來說是無法承受的。

## 在 Founder's Playbook 中（[AI-Native Startup Lifecycle](/articles/ai-native-startup-lifecycle)）

該手冊將 MCP 視為每個階段的主要整合機制：

- **構想階段** ——Cowork 使用 Gmail 和 Google Calendar MCP 來管理外展信件、安排客戶訪談、執行第 7 天追蹤。
- **MVP 階段** ——「在構想階段管理探索後勤的同一組 MCP 整合在此同樣適用」，用於回饋會議排程、錯誤報告分類、迭代週期追蹤。
- **規模化階段** ——與**你的競爭對手聞所未聞的利基產業系統**進行 MCP 整合被列為護城河組成要素（例如，通用型醫療帳務 AI 在 340B 藥品計畫索賠上會出錯；垂直專精且接入 MCP 的競爭對手則不會）。

手冊中兩個案例研究具體說明了 MCP 作為護城河的論點：
- **Kindora** 發布了一個 MCP 連接器，讓非營利組織可以*在 Claude 內部*存取其潛在客戶開發工具——產品是透過 MCP 被消費的，而不僅僅是*與* MCP 整合。
- **Anthropic Skills** 被引用為重複性工作流程的編纂介面（「我如何審計商業租約」、「我如何分類病患入院表單」）——Skills + MCP + 記憶共同構成了 [Compounding Data Moat](/articles/compounding-data-moat) 概念所描述的專有基底。

Computer use 在手冊本身中較不突出，但 Cowork 被列為跨「每個階段」運行的營運層——而 Cowork 正是 computer use 覆蓋 MCP 未及之處的地方。

## 與 harness 縮減的關聯

[Harness Shrinkage as Models Improve](/articles/harness-shrinkage-as-models-improve) 預測提示詞腳手架、權限和驗證邏輯會隨著模型改進而向內遷移。MCP 和 computer use 是 harness 的*反面*——它們是模型與世界*之間*的連接器。它們不會縮減；它們會變得*更廣*（更多系統、更多介面）且*更快*（每次動作的延遲更低）。縮減的邊界是模型工具選擇決策*周圍*的 harness，而非工具集本身。

附帶說明：隨著模型越來越擅長判斷何時使用 computer use、何時要求真正的 MCP，今天許多手動撰寫 MCP 伺服器的工作可能會變成「請模型建構你需要的連接器」。這仍然不是 harness——更像是模型撰寫的基礎設施。

## 與 [Agentic Misalignment (AM)](/articles/agentic-misalignment) 及問責制的關聯

MCP 和 computer use 正是將 LLM 轉變為[能夠採取重大行動的 agent](/articles/agentic-misalignment) 的基底。兩者都將模型的觸及範圍延伸至：
- 客戶的 CRM
- 客戶的電子郵件
- 客戶的行事曆
- 最終，客戶的整個桌面

[Human-AI Accountability Redesign](/articles/human-ai-accountability-redesign) 的「決策權」子前線正是治理此事的機制——agent 透過 MCP/computer use 自主執行什麼，vs. 什麼需要明確的人類核准。[Claude Code Auto Mode](/articles/claude-code-auto-mode) 是一個具體實例：分類器自動核准安全的 MCP/工具呼叫，阻擋有風險的呼叫。

## 開放問題

- MCP 生態系的成長速度 vs. computer use 的品質曲線：在什麼時間點 computer use 會變得*夠好*，以至於建構 MCP 伺服器的邊際價值下降？Boris 暗示這還要好幾年，但未量化。
- Computer use 是可持續的介面還是過渡技術？如果大多數知識工作軟體在未來 24 個月內加入 MCP 支援，computer use 的角色將縮減至僅限舊系統/桌面系統。
- MCP 安全模型：當手冊建議為獨立創辦人將 MCP 接入 Salesforce、Gmail、Calendar 時，攻擊面隨採用率擴大。在已攝取的任何來源中均未討論。
- Cowork 的 computer use 防護機制與 Claude Code 的 auto-mode 分類器相比如何？不同的部署情境，可能有不同的風險特徵。
