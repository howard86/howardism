## 摘要

[Thariq Shihipar](/articles/thariq-shihipar) 對 [HTML 計畫](/articles/html-as-the-new-markdown) 一個顯而易見的反對意見的回應：HTML 不像 markdown 那樣容易編輯，那麼當你想修改一張複雜的表格、或一組埋在 HTML 產出物裡的規則時，該怎麼辦？你**請 Claude 打造一個臨時、拋棄式的使用者介面——一個微應用——專門用來編輯計畫中那一個特定部分。** 與其手動編輯產出物或在聊天中反覆磨蹭，你直接生成*為這次編輯量身打造的理想工具*，並只用一次。

> 「這甚至已經不算是個人軟體了。這像是子……像是建立在微軟體之上的微軟體。」

## 實際案例

Thariq 的 HTML 實作計畫包含一組**決策規則**，用於把不同的 CSV 資料類型轉換成視覺化呈現。他覺得這些規則「有點隨意」，想要進一步打磨——這在純文字或原始 HTML 中改起來相當繁瑣。提示詞如下：

```
I want to create an edible HTML artifact to help me define the decision rules...
I don't like the ones we have right now. Make this a custom UI that helps me with
structure but gives me flexibility. Design the ideal interface for this problem.
```

（「edible」=「editable」——又是一個模型輕鬆處理掉的無傷大雅的錯字。）Claude 回傳了一個專門打造的網頁：輸入欄位、下拉選單、新增／移除按鈕，以及**一顆把最終設定以 markdown 格式複製回剪貼簿的按鈕。** Thariq 用結構化、視覺化的方式編輯這些規則，得到正確的輸出，然後把雕琢過的資料帶回他的主要工作流程。

## 兩個讓它成立的設計要點

- **「為這個問題設計理想的介面。」** 人類指定的是*問題*，而不是元件組合，並讓模型自己挑選介面——約束加上自由度，與 [HTML as the New Markdown](/articles/html-as-the-new-markdown) 中相同的提示手法。
- **複製回 markdown。** 這個微應用是覆蓋在持久文字之上的暫時性*編輯介面*。它修補了 HTML 最大的弱點（diff／版本控管）——真實來源透過 markdown 進行往返流通，而豐富的 UI 只在編輯期間存在。

## 為什麼這現在合理：豐裕思維

在稀缺經濟學的觀點下，為了一個編輯決策而啟動一支只用一次的應用會顯得荒謬。但在 [Compute Allocator](/articles/compute-allocator) 的**豐裕思維**下卻是合理的：生成已經便宜到能讓拋棄式工具成為對自身流程的合理投資。「我們現在已負擔得起生成這些拋棄式工具，讓自己的流程更有效率、也更愉悅。」這就是 [Printing Press Software Democratization](/articles/printing-press-software-democratization) 在單一任務尺度上的展現——軟體便宜到變成一次性的方便品，而不是專案。

## 待解決的問題

- 拋棄式微應用與工具氾濫之間的界線在哪裡？如果每次編輯都生出一支量身訂做的 UI，工作流程會不會碎裂？
- 複製回 markdown 的往返流通能否從設定形態的資料（規則、表格）擴展到更豐富的產出物？
- 這些微應用能否被模板化／重複使用，而非每次都重新生成——而到了哪個臨界點，這就反而背離了「拋棄式」的框架，轉而變成 [durable tooling](/articles/living-design-system)？
