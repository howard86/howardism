## 摘要

[Andrej Karpathy](/articles/andrej-karpathy) 在 2025 年創造了「vibe coding」一詞，一年後他為其正式的後繼者命名：**agentic engineering**。兩者的區別在於*哪條線在移動*。Vibe coding **提升了門檻下限** — 現在任何人都能開發軟體。Agentic engineering **在維持專業軟體品質標準的同時大幅加速**：「你不能因為 vibe coding 就引入漏洞；你仍然要為你的軟體負責，但你能不能更快 — 而且如何正確地做到這一點？」這是一門工程學科，用於協調能力參差不齊、容易出錯、具有隨機性但強大的 agents，同時不犧牲品質。

## 兩條線

- **Vibe coding — 下限上移。** 每個人都能 vibe-code 任何東西。「太棒了，不可思議。」民主化（參見 [Printing Press Software Democratization](/articles/printing-press-software-democratization)）。重點不是品質，而是可及性。
- **Agentic engineering — 上限上移，品質不變。** 你保留專業軟體的責任（安全性、正確性、可維護性），並利用 agents 加速，*同時不*低於該標準。「正確且妥善地做到這一點，就是 agentic engineering 的領域。」

這是兩種不同的活動，而非同一條線上的兩個點。一個降低了進入成本；另一個為已經達標的人提升了產出上限。

## 「10 倍不是加速幅度」

Karpathy 明確地將舊有的「10 倍工程師」說法視為*太小*而予以淘汰：「10 倍不是你獲得的加速幅度……非常擅長這件事的人，峰值遠超 10 倍。」Agentic engineering 能力的上限非常高，而平庸與 AI 原生從業者之間的差距在擴大，而非縮小。（呼應 [Harness Shrinkage as Models Improve](/articles/harness-shrinkage-as-models-improve)：隨著模型改進，槓桿持續增長；約束瓶頸變成操作者的品味 — 參見 [Outsource Your Thinking, Not Your Understanding](/articles/outsource-thinking-not-understanding)。）

## AI 原生從業者的樣貌

當被問到平庸使用者與完全 AI 原生的 cloud code / codex / open claw 使用者有何不同時，Karpathy 的回答平凡卻重要：**投資你的設定，使用工具的所有功能。** 就像當年從 Vim 或 VS Code 中獲得最大效益的工程師一樣 — 現在應用到 Claude Code / Codex 上。精通在於對配置和功能的熟練度，而非某個秘密 prompt。

## 招聘必須重構

一個實際的推論：大多數團隊仍然用舊範式招聘（解題、LeetCode）。Karpathy 主張 agentic engineering 的招聘應該像是*「給我一個真正大型的專案，看某人如何好好實作它」* — 例如，建構一個安全的 Twitter-clone-for-agents，然後一個紅隊 agent（「codex 5.4 xhigh」）嘗試攻破它但*無法成功*。招聘應該測試可驗證的端到端建構與防禦能力，而非孤立的解題能力。（參見 [The Verifiability Thesis](/articles/verifiability-thesis) 了解為何「而且它無法被攻破」才是承重的那一半。）

## 人類的殘留角色

即使在高上限下，人類仍然掌控**規格、品味、判斷與監督** — agents 負責填空。他的 MenuGen 戰爭故事：agent 用*電子郵件地址*而非持久性使用者 ID 來匹配 Stripe 和 Google 帳戶 — 「這麼奇怪的做法」，正是 [Jagged Intelligence (Ghosts, Not Animals)](/articles/jagged-intelligence) 所預測的那種錯誤。你必須設計規格（「這些必須是我們用來綁定一切的唯一使用者 ID」）並提供品味；agent 處理你已經不再記憶的 API 細節。

## 開放問題

- Karpathy 暗示有「一個非常[有價值的]領域」適合創辦人，但不願透露是哪個（不想「在台上含糊發文」）。他所指的是哪個可驗證的 RL 環境領域？
- 如果平庸者與 AI 原生者之間的差距持續擴大，這對團隊組成意味著什麼 — 少數極端異常值加上 agents，還是廣泛的中階人員配置？
