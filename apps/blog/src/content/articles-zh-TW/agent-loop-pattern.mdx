## 摘要

**迴圈（loop）** 是一種 agent 流程，反覆執行一個 prompt 直到佇列清空或達到停止條件。截至 2026 年中，三個趨同的實作指向迴圈正在成為與單次 session 同等地位的原語：Anthropic 的 `/loop` 斜線指令（cron 排程、重複執行）、Anthropic 的 **routines**（伺服器端的 `/loop`），以及 Matt Pocock 的 **Ralph Wiggum loop**（bash + `claude --permission-mode accept-edits` 放在 `while` 中）。[Boris Cherny](/articles/boris-cherny) 稱迴圈為「未來」；[Matt Pocock](/articles/matt-pocock) 將其作為端到端工作流程中 AFK 骨幹。

## 兩大迴圈家族

### Cron 排程迴圈（`/loop`、routines）

用於 [Claude Code](/articles/claude-code) 和 [Cowork](/articles/cowork)。機制：agent 透過工具呼叫 `cron` 在未來某個時間排程一個任務；該任務在指定時間重新進入 agent 並附帶執行指令。排程可以重複（每分鐘、每 5 分鐘、每天）。

Boris Cherny 報告的用途：
- 看顧 PR — 修復 CI、自動 rebase
- 維持 CI 健康 — 修復不穩定的測試
- 每 30 分鐘彙整 Twitter 回饋
- 「隨時都有數十個迴圈在運行」
- 過夜：「數千個 agents」進行更深度的工作

**Routines** 是同一原語的伺服器端版本，因此即使筆電關閉也能持續運行。

### 消化待辦事項迴圈（Ralph Wiggum loop）

由 [Matt Pocock](/articles/matt-pocock) 等人使用。機制：一個 shell 腳本以固定 prompt 執行 agent，prompt 指示它從待辦事項中挑選下一個任務並完成，腳本隨後重新啟動。待辦事項是一個目錄中的 markdown issue 檔案（或 GitHub issues）。

Pocock 的 `once.sh` 骨架：
```bash
issues=$(cat issues/*.md)
recent_commits=$(git log -5 --oneline)
prompt=$(cat prompt.md)
claude --permission-mode accept-edits "$prompt" --context "$issues" "$recent_commits"
```

「迴圈」包裝器只是重複執行 `once.sh`，直到 agent 發出哨兵訊號（`no more tasks`）或 harness 將其停止。

Prompt 強制執行 **AFK 限定的任務選擇** — 只有標記為 `AFK`（相對於 `human-in-loop`）的任務才有資格被選取。

## 為什麼迴圈重要

1. **將規劃攤提到多次執行中。** 一次仔細的規劃 session（例如透過 [Design Concept Grilling](/articles/design-concept-grilling)）建立一個 Kanban 待辦事項（參見 [Vertical Slice Tracer Bullets](/articles/vertical-slice-tracer-bullets)）；迴圈在無需進一步人工介入的情況下消化它。
2. **數小時的任務變得可行。** 迴圈不使用一個巨大的 context window，而是將工作切分為多個全新的 session — 每次都保持在 [Context Window Smart Zone](/articles/context-window-smart-zone) 內。
3. **平行化。** 獨立的待辦事項在各自的沙箱中同時執行。Pocock 的 **Sandcastle** 函式庫透過在 Docker 容器中為每個 issue 建立 git worktree 來實現；合併 agent 事後進行調和。
4. **閒置算力。** Boris 的過夜設定是在便宜的閒置時段運行千個 agent 迴圈 — 這些工作不值得人類花一個晚上來做，但以 agent 成本來看能產生價值。

## AFK 與 human-in-loop 任務

[Matt Pocock](/articles/matt-pocock) 的關鍵區分：

- **AFK 任務** — 實作、重構、測試鷹架、文件維護、CI 修復。Agent 無需逐步核准即可成功；驗證是自動的（測試、型別、linter）。
- **Human-in-loop 任務** — 對齊、設計選擇、優先排序、QA。這些沒有機械式驗證；需要品味和隱性脈絡。

迴圈適用於 AFK 類別。嘗試對 human-in-loop 工作進行迴圈會產生偏移 — agent 做出看似合理但錯誤的決定並不斷累積。

## 驗證是天花板

Pocock 更強的主張：**回饋迴圈的品質決定了迴圈能做到什麼程度的天花板。** 沒有好的測試、型別和 linter，迴圈就是「盲目寫程式」。這與 [Agent Harness Engineering](/articles/agent-harness-engineering) 中關於機械式強制執行的觀點相同 — 迴圈只是更鮮明地暴露了成本，因為沒有人類來攔截偏移。

## 與模型發展軌跡的關聯

[Boris Cherny](/articles/boris-cherny) 報告 Opus 4.7 *自發地*在未被提示的情況下啟動迴圈：

> 「我會告訴它，『去拉這個資料查詢。』然後它說，『嘿，我注意到資料隨時間在變化。我會啟動一個迴圈，每 30 分鐘給你一份報告。』」

這符合 [Harness Shrinkage as Models Improve](/articles/harness-shrinkage-as-models-improve) — 過去需要 harness 注入的能力變成了模型的自然行為。迴圈原語仍然存在，但使用者不再需要主動呼叫它。

## 開放問題

- 當模型自行排程迴圈（4.7 行為）時，誰擁有預算？Boris 回答「模型自己決定」— 但這將成本紀律推入模型的訓練中，而非 harness。
- 具有足夠聰明模型的迴圈是否仍需要 Kanban 待辦事項，還是模型會從原始目標中自行選擇下一個任務？
- 迴圈產出審查現在是 [Matt Pocock](/articles/matt-pocock) 坦承的瓶頸 — 「我們只需要準備好做更多的 code review。」
