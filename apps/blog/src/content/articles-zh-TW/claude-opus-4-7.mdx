## 摘要

Claude Opus 4.7 是 Anthropic 正式發布的前沿模型，作為 Opus 4.6 的直接升級（相同定價：$5/M 輸入、$25/M 輸出；模型 ID `claude-opus-4-7`）。它在進階軟體工程、字面指令遵循、高解析度視覺和檔案系統記憶方面有所提升，但整體能力仍不及限量發布的 [Claude Mythos Preview](/articles/llm-driven-vulnerability-research)。它是首個搭載 Mythos 級網路安全防護的模型，採用 [Project Glasswing](/articles/llm-driven-vulnerability-research#Project%20Glasswing) 技術。

## 細節

### 相較 Opus 4.6 的能力差異

- **最困難任務的軟體工程**：明確以「交付你最困難的程式碼工作」為行銷定位。在 Finance Agent、GDPval-AA 上達到 SOTA；在 SWE-bench Verified/Pro/Multilingual 上有所提升（排除記憶化標記問題後改善仍然成立）。
- **指令遵循——字面化**：顯著更加字面化。Anthropic 警告，為早期模型調整的提示詞「現在有時會產生意外結果」，因為 Opus 4.7 不再跳過或寬鬆解讀部分內容。**重新調整是必要的遷移步驟**，而非可選。
- **多模態**：接受長邊最高 2,576 px 的圖片（約 3.75 MP，超過先前 Claude 模型的 3 倍）。支援密集截圖閱讀（computer-use）、複雜圖表提取、像素精確引用。這是模型層級的變更，非 API 參數。
- **檔案系統記憶**：在長期多 session 工作中更善於使用檔案系統支援的記憶；後續任務需要更少的前置上下文。
- **安全性**：整體安全概況與 4.6 相似。在誠實度和**提示注入抵抗力**方面更好；在管制物質的過度詳細減害建議方面略弱。「大致上對齊且值得信賴，但尚未完全理想。」根據 Anthropic 的評估，Mythos Preview 仍是對齊程度最高的模型。

### Token 經濟變化（遷移風險）

兩個複合效應增加了 token 消耗：

1. **更新的 tokenizer**：相同輸入映射為 **1.0–1.35× 更多 token**，取決於內容類型。
2. **在較高努力等級下思考更多**，特別是在 agentic 設定的後續回合中——產生更多輸出 token。

Anthropic 聲稱在其內部程式碼 eval 中，跨努力等級的淨效果是有利的，但明確建議在實際流量上進行測量。使用者可透過 `effort` 參數、任務預算或明確的簡潔提示來應對。這直接影響 [Claude Code Best Practices](/articles/claude-code-best-practices) 中 context window 作為主要限制的主題；交叉參考 [Scale-Dependent Prompt Sensitivity](/articles/scale-dependent-prompt-sensitivity) 中的簡潔限制發現。

### 努力等級

引入新的 `xhigh`（「extra high」）努力等級，位於 `high` 和 `max` 之間。權衡面向：困難問題上的推理深度 vs. 延遲/token。

- **Claude Code 預設在所有方案中提升至 `xhigh`。**
- Anthropic 建議程式碼/agentic 使用從 `high` 或 `xhigh` 開始。

### 網路安全能力與防護

- Opus 4.7 是**首個 Project Glasswing 後模型**，搭載「自動偵測並阻擋指示禁止或高風險網路安全用途的請求」的防護措施。
- 網路安全能力在**訓練期間被差異化降低**（不僅僅是推論時過濾）。
- 在網路安全方面仍不及 [Mythos Preview](/articles/llm-driven-vulnerability-research)；CyberGym 分數已更新（harness 改善將 Opus 4.6 基線從 66.6 → 73.8）。
- 合法安全研究人員（漏洞研究、滲透測試、紅隊演練）透過新的 **Cyber Verification Program** 而非預設存取進行路由。

這直接實現了 [LLM-Driven Vulnerability Research](/articles/llm-driven-vulnerability-research#Project%20Glasswing) 中聲明的路線圖承諾：「即將推出的 Claude Opus 模型將搭載針對 Mythos 級輸出開發的新防護措施。」

### 同步發布

- **任務預算（公開測試版，API）**：開發者引導的 token 支出分配，跨越較長的執行——這是 [Client-Side Agent Optimization](/articles/client-side-agent-optimization) 組合空間中預算槓桿的伺服器端類比。
- **`/ultrareview`** Claude Code 中的斜線命令：專用審查 session，讀取變更並標記錯誤/設計問題。Pro 和 Max 使用者獲得三次免費 ultrareview。
- **[Auto mode](/articles/claude-code-auto-mode) 擴展至 Max 使用者**（先前僅限 Team 的 research preview）。

### 可用性

- 所有 Claude 產品、Claude API、Amazon Bedrock、Google Cloud Vertex AI、Microsoft Foundry。
- API 模型 ID：`claude-opus-4-7`。
- 定價與 Opus 4.6 相同。

## 開放問題

- Hakim（2026）在 Opus 4.6 上的簡潔限制發現是否能在 Opus 4.7 上複製，還是字面指令遵循改變了彈性？具體而言：`<50 words` 是否仍能在 GSM8K 上產生 +13.1pp？
- Opus 4.7 在 HotpotQA 風格的組合掃描中是否仍作為規劃者表現不佳，還是改善的指令遵循縮小了 AgentOpt（Hua et al., 2026）所識別的差距？
- 典型 Claude Code session 中的實際 token 膨脹倍數是多少（1.0–1.35× 取決於內容——程式碼密集 vs. 散文密集輸入的分佈為何）？
- `xhigh` 在程式碼 evals 上與 `max` 相比如何？遷移指南說「從 high 或 xhigh 開始」——`max` 對程式碼是否曾經值得？
- 在字面指令遵循下，現有 CLAUDE.md / system prompt 中有多少比例的防護性措辭變得適得其反？
