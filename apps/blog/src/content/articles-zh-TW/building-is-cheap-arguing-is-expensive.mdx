## 摘要

[Fiona Fung](/articles/fiona-fung) 在 agentic coding 使生成幾乎免費之後，為技術辯論訂下的規則：**「在技術辯論中，程式碼說了算——建造成本低，爭論成本高。」** 當你能在預約白板會議的時間內生成三個真實實作時，你就不再*爭論*方案，而是開始*比較*方案。在 [Claude Code](/articles/claude-code) 上的推論規範：用原型加 dogfood 取代撰寫設計文件，並減少深度規劃文件，改以 PR 和原型中的討論取代。

## 產生三個 PR 的做法

Fung 的實戰故事：在導入 Claude Code 時，她和 [Boris Cherny](/articles/boris-cherny) 就一個重構方案展開辯論。舊有反射：「我們去會議室白板討論吧。」取而代之的是：*「多虧了 Claude，我產生了三個不同版本的 PR」*——這讓他們有了更好的技術辯論，因為她不僅能比較程式碼的實作方式，還能比較**對呼叫端的影響**。產出物（真實、可執行的 PR）比論點（白板上的抽象概念）承載更多資訊，而且現在產出它的成本很低。

## 原型優先於設計文件

同樣的經濟學改寫了原型開發。過去，原型開發將團隊分成兩個陣營——*快速回饋*與*偷工減料，最終被迫交付無法擴展的拋棄式程式碼*。有了 Claude，第二個反對意見被削弱了：「原型開發是很好的起步方式，因為我們可以迭代學習，然後更快地將原型擴展到生產環境。」所以 Claude Code 的規範是：有想法 → 做原型 → dogfood 它 → 然後再決定。（參見 `prototype`「拋棄式原型以充實設計」的方法。）

## 被減少的東西：設計文件

團隊*減少*的東西：「非常深入的規劃和設計文件。我們大部分的討論實際上發生在 PR 或原型中。」這是 [Verification as the New Bottleneck](/articles/verification-as-the-new-bottleneck) 的下游效應——當工程頻寬不再稀缺時，為了*保護*該頻寬而存在的大量前期規劃就失去了正當性。重構和架構清理——過去總是在「什麼時候才有時間？」的預算爭奪中落敗——也不再是瓶頸。

## 需要把握的張力

這個規範與 wiki 中規劃優先的概念存在建設性的張力。[Design Concept Grilling](/articles/design-concept-grilling)（[Matt Pocock](/articles/matt-pocock)）和 [Vertical Slice Tracer Bullets](/articles/vertical-slice-tracer-bullets) 主張你必須在建造之前達成共享的*設計概念*。調和之道：「建造成本低」並不廢除設計思維——它重新定位了設計思維。你仍然需要品味和規格（[Outsource Your Thinking, Not Your Understanding](/articles/outsource-thinking-not-understanding)），但你是透過*比較已建造的產出物*而非*在抽象層面爭論*來達成它們。低成本的建造使原型成為設計的媒介，而非設計的替代品。（Karpathy 在 [Vibe Coding vs. Agentic Engineering](/articles/vibe-coding-vs-agentic-engineering) 中取得了同樣的平衡：人類仍然掌握規格。）

## 開放問題

- 「產生三個再比較」何時變得浪費——在什麼決策權重下，真正的爭論（或設計文件）仍然比三個實作更便宜？
- 如果設計討論發生在 PR/原型中，*理由*記錄在哪裡供未來讀者參考——「我們為什麼選擇這個」的知識是否能存活，還是它與 [Code as Source of Truth](/articles/code-as-source-of-truth) 有著同樣的過時問題？
