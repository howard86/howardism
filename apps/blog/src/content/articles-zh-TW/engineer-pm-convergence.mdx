## 摘要

[Boris Cherny](/articles/boris-cherny)（Sequoia AI Ascent 2026）和 [Cat Wu](/articles/cat-wu)（Lenny's Podcast，2026 年 4 月）都從 [Anthropic](/articles/anthropic) 內部報告了相同的趨勢：角色正在融合。工程師做 PM 的工作，PM 寫程式，設計師寫程式，[Claude Code](/articles/claude-code) 團隊中每個職能角色都在寫程式。在這個融合角色中，真正重要的共通技能是**產品品味**——在建造成本已經很低的情況下，決定要建造什麼。這挑戰了傳統的跨職能團隊模式（一個 PM、一個設計師、數名工程師、獨立的 eval/QA），並指向由高度掌握脈絡的通才組成的更小團隊，每個人獨立負責端到端的交付。

## Anthropic 內部實際發生的事

來自 [Cat Wu](/articles/cat-wu)：

- **[Claude Code](/articles/claude-code) 團隊中每個人都寫程式**——工程經理、產品經理、設計師、資料科學家、財務、使用者研究員。
- 設計師以前是前端工程師。
- 「我們團隊中有很多工程師完全有能力端到端地從在 Twitter 上看到使用者回饋，到在週末前交付一個產品，幾乎不需要產品團隊的介入。」
- 招聘偏好：「我們非常專注於招聘具有優秀產品品味的工程師。」
- Boris ↔ Cat 的分工是「80% 心靈感應，20% 領域驅動。」

來自 [Boris Cherny](/articles/boris-cherny)：

- 「跨領域通才」——同時擅長設計的工程師，或產品加資料科學。
- 「我們團隊中每個人都寫程式。」
- 「產品工程師」原型（iOS + web + server）是*基本*通才；新模式是**跨學科**通才。

## 品味論點

> 「隨著寫程式變得越來越便宜，決定要寫什麼變得更有價值。」

— Cat Wu

程式碼生成成本正在下降。瓶頸向上游轉移：
- 從一萬個 GitHub issues 中挑選要交付哪個功能
- 設計 UX 使模型的優勢被展現、弱點被彌補
- 知道何時以 research preview 推出，何時作為完整產品發布
- 區分人們*說*他們想要的 90 個功能和他們*實際會用*的 10 個

這就是產品品味，它不綁定於職稱——工程師可以有，設計師可以有，PM 可以有。有品味的人交付得好；沒有的人交付出漂移。

## 為什麼是現在（Lenny 的對比觀點）

Lenny 在 Cat 的訪談中提出反駁，引用 Anthropic 成長負責人 Amol Avasare 之前的節目：Amol 說他需要*更多* PM，因為工程師交付速度太快，設計師和 PM 跟不上——每天都有新功能。

Cat 同意這*可能*是情況，但在 Claude Code 上更偏好具有品味的工程師路線：招聘更少這樣的人，擴大槓桿，減少協調交接。

這兩個答案並不矛盾——它們描述的是不同的團隊形態：
- **Cat 的模式：** 高度自主的小型通才團隊（Claude Code，約二十幾人）
- **Amol 的模式：** 較大的組織，工程師動作快，需要 PM/設計支援來保持同步（Growth、GTM、Enterprise）

## 不對稱性：人類 EQ 仍然存在

Cat 指出什麼沒有融合：隱性的、常識性的、高 EQ 的工作——知道與利害關係人溝通的正確場合、感知發布何時準備就緒、知道什麼算是公平的取捨。模型在這方面正在進步但尚未到位，人類仍然提供跨發布的連接組織。

## 「直接去做」

Cat 的人生座右銘：

> 「工作是虛構的。如果你理解約束條件，你就能弄清楚你能做什麼，然後就快速去嘗試，從錯誤中學習，如果做錯了就道歉或修正。」

這是使角色融合得以運作的*文化*基底。如果角色被職位描述所限制，沒有人會跨職能行動。如果「做需要做的事」是常態，融合就會自然發生。

使命對齊是讓「直接去做」不會混亂的關鍵——Cat：「如果有兩個競爭的優先事項，我們會討論哪個對 Anthropic 的使命更重要。這讓決定優先哪個變得容易得多。然後每個人都會支持我們決定的那個。」

## 啟示

1. **招聘看品味，不看角色。** Cat 明確的標準：任何能展示強大產品品味的人。
2. **積極交叉培訓。** 如果你的設計師不寫程式，那是組織選擇，不是限制。
3. **更小的團隊。** 一個每人都負責端到端的 5 人團隊，比一個有交接的 15 人團隊交付更快。
4. **PRD 變得更輕量。** Cat：對模糊功能用簡短的要點式 PRD；指標報告和團隊原則完成大部分對齊工作；完整 PRD 只用於重基礎設施專案。
5. **職涯階梯碎片化。** Cat：「我們犧牲了產品一致性」作為交付速度的取捨；職涯階梯的一致性是更安靜的犧牲品。

## 開放問題

- 這能擴展到超過約 50 人的 Claude Code 式團隊嗎？Boris 保留態度：「我認為這將是一個持續數年的問題。」
- 在工程師做 PM 工作的公司中，正式的 PM 職涯階梯會怎樣？在 Anthropic 這仍是開放問題，據 Cat 所述。
- 跨學科通才是一個招聘門檻——供給從哪裡來？轉職者，還是新鮮人偏向 AI 原生教育？
