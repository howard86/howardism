## 摘要

[Fiona Fung](/articles/fiona-fung) 對 [Claude Code](/articles/claude-code) 提出的「辛辣」組織變革：**每位主管都先從 IC 做起**，並持續直接負責產品的一部分。組織保持「agile 且盡可能扁平」。讓這件事變得可行的關鍵在於，agentic coding **消除了上手成本**——過去正是這個成本讓主管重返程式碼庫變得令人卻步——因此主管可以重新奪回 maker 時間，而不需要經歷痛苦的學習曲線。

## 這項變革

當 Fung 加入 Claude Code 時，她做的第一件事之一就是與招募團隊合作，確保**每位主管在承擔人員管理責任之前，都先以 IC 身份開始工作**。理由是：這讓主管能「捲起袖子、深入程式碼庫，體會身為團隊工程師的感受」，然後才承擔支持團隊成員的責任。她將扁平組織加上主管擁有 IC 職責，列為她優先推動且「效果非常好」的三件事之一。

## 為何是現在：上手不再令人卻步

Fung 在職涯中多次在主管與 IC 角色之間切換（Meta、Microsoft、Anthropic），每次回到 IC 工作時總會感到「些許猶豫」——「如果你有一陣子沒做了，這可能會讓人卻步。」Agentic coding 消除了這個摩擦：加入 Claude 的上手過程是「截然不同的體驗」，因為她可以請 Claude 教她了解系統的各個面向，而不需要佔用工程師的時間（[Code as Source of Truth](/articles/code-as-source-of-truth)）。過去將主管*推離*程式碼庫的成本——學習時間以及開口求助的社交成本——大幅下降。

## Dogfooding 的理由

更深層的原因是 dogfooding（見 [Dogfooding as Product Discipline](/articles/dogfooding-as-product-discipline)）：「在 Claude Code 團隊，我們用 Claude 來打造 Claude，用 Claude Code 來打造 Claude Code」——所以一位不在程式碼庫裡的主管，就無法體驗他們正在交付的產品。Managers as ICs 正是領導層保持「從骨子裡感受產品」的方式。這是在管理層面對 dogfooding 規範的結構性強制執行。

## 與工程師–PM 趨同的關聯

這是 [Engineer PM Convergence](/articles/engineer-pm-convergence) 在管理層的體現：[Boris Cherny](/articles/boris-cherny) 指出 Claude Code 團隊的*每一位*成員都寫程式（EM、PM、設計師、資料科學家、財務、使用者研究員）。Managers as ICs 將「人人寫程式」延伸到組織架構的上層，並與 Anthropic 運行的更輕量、更扁平的 [AI Native Product Cadence](/articles/ai-native-product-cadence) 一致。

## 開放問題

- Fung 自己的開放問題：「你還需要分開的 iOS 和 Android 組織嗎？」——如果工程師能透過 Claude 跨平台靈活切換，傳統的平台分割組織可能也會瓦解。扁平化能走多遠？
- 主管即 IC 的模式能否在組織規模超過一定程度後依然適用，還是只在 Claude Code 團隊規模小、程式碼庫對 Claude 可讀的情況下才行得通？
