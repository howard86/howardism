## 摘要

[Fiona Fung](/articles/fiona-fung) 在 [Claude Code](/articles/claude-code) 推動的「辛辣」組織變革：**每位經理都先從 IC 做起**，並持續直接負責產品的一部分。組織維持「敏捷且盡可能扁平」。如今之所以可行，是因為代理式程式設計**大幅降低了入職成本**——過去這項成本會讓經理重新進入程式碼庫變得令人望而生畏；現在經理可以重新取回創作者時間，不必經歷痛苦的適應期。

## 這項變革

Fung 加入 Claude Code 後，最先做的事情之一，就是與招募團隊合作，讓**每位經理在承擔支援人員的責任前，先從 IC 做起**。理由是：這讓經理能在承擔支援團隊成員的責任之前，「捲起袖子、進入程式碼庫，了解成為團隊工程師是什麼感覺」。她將扁平組織加上經理擁有 IC 責任，視為自己優先推動且「效果非常好」的三件事之一。

## 為什麼是現在：入職不再令人望而生畏

Fung 的職涯曾在經理與 IC 角色間切換（Meta、Microsoft、Anthropic），每次回到 IC 工作時，她總會感到「些微的遲疑」——「如果你有一段時間沒做，這可能會令人望而生畏」。代理式程式設計消除了這種摩擦：加入 Claude 的體驗「非常不同」，因為她可以請 Claude 教她了解整個表面範圍，而不是消耗工程師的時間（[程式碼即真相來源](/articles/code-as-source-of-truth)）。過去把經理**推離**程式碼庫的成本——適應時間，以及開口求助的社交成本——大幅下降。

## 以身作則使用自家產品的理由

更深層的原因是 dogfooding（參見[將自家產品使用視為產品紀律](/articles/dogfooding-as-product-discipline)）：「在 Claude Code，我們用 Claude 建造 Claude，也用 Claude Code 建造 Claude Code」——因此，不在程式碼庫裡的經理，並沒有體驗自己正在交付的產品。身兼 IC 的經理，是領導層持續「從骨子裡感受產品」的方式。這是在管理層落實 dogfooding 規範的結構性手段。

## 與工程師–PM 融合的關聯

這是[工程師與 PM 融合](/articles/engineer-pm-convergence)在管理層的體現：[Boris Cherny](/articles/boris-cherny) 指出，Claude Code 團隊的*每一位*成員都會寫程式（EM、PM、設計師、資料科學家、財務、人員研究員）。身兼 IC 的經理將「人人寫程式」延伸到組織圖的更上層，也符合 Anthropic 運作的更輕量、更扁平的[AI 原生產品節奏](/articles/ai-native-product-cadence)。

## 待解決的問題

- Fung 自己提出的問題：「你是否仍需要分開的 iOS 與 Android 組織？」——如果工程師能透過 Claude 跨平台彈性工作，傳統的平台分割組織也可能隨之消解。扁平化究竟能走多遠？
- 經理兼任 IC 是否能擴展到某個組織規模之後，還是只有在 Claude Code 規模較小、且程式碼庫容易被 Claude 理解時才可行？
