## 摘要

[Fiona Fung](/articles/fiona-fung) 為 AI 原生組織重新設計的知識共享方式：**當編碼頻寬很高時，文件過時的速度比任何人能維護的速度都快——因此程式碼庫成為唯一真相來源，任何你希望保持正確的東西都要簽入程式碼庫。** 規格變成提交到 repo 的技能；新人引導透過請 Claude 教你相關的表面積來完成，而不是把工程師拉進技術深潛會議。「我們的程式碼就是我們的唯一真相來源。」

## 為什麼文件在 AI 原生組織中會腐化

這個機制是 [Verification as the New Bottleneck](/articles/verification-as-the-new-bottleneck) 的直接後果：更高的編碼產出意味著程式碼變化更快，因此任何*不在更新迴圈內*的文件幾乎會立即過時。Fung 的處方：「無論你的唯一真相來源是什麼——不管是規格——把它變成一個簽入程式碼庫的技能，這樣你就能保持它的時效性。」Repo 是唯一保持最新的產出物，因為它就是被變更的那個東西。

## 規格簽入 repo 使規格漂移驗證成為可能

將規格簽入程式碼庫不僅僅是保鮮——它讓機械式驗證成為可能：「Claude 非常擅長驗證規格漂移。」一個已提交的規格既是人類可讀的意圖，*也是*審查 agent 用來比對差異的機器可檢查參考。（這與 Symphony 的 `WORKFLOW.md` 和 Claude Code 的 `CLAUDE.md` 是同一招——repo 版本化的純文字作為控制平面；參見 [Symphony](/articles/symphony)、[Claude Code Best Practices](/articles/claude-code-best-practices)。）

## 透過 Claude 引導新人，而非透過同事

Fung 自己加入 Claude Code 時的引導方式：她不是（只）跟工程師做技術深潛，而是先*跟 Claude* 做第一次深潛——「在我開始修這個 bug 之前，你能教我這個 bug 周圍的表面積和相關區域嗎？」她報告的效果：**新人上手時間縮短，對*其他*團隊成員的成本也降低**——新加入者不再佔用資深工程師的時間來獲取上下文。作為管理者，這也讓她能無愧疚感地重新進入程式碼庫（「我不覺得我在浪費任何人的時間」）。程式碼庫加上 Claude *就是*引導文件。（參見 [Managers as ICs](/articles/managers-as-ics)。）

## 與 CLAUDE.md 和 agentic 債務的關係

Code as Source of Truth 是正面綱領；[Agentic Technical Debt](/articles/agentic-technical-debt) 是它所防範的失敗模式。持久化、已提交的上下文（CLAUDE.md、簽入的技能/規格）阻止每個 agentic session 重新推導架構決策。本 wiki 自身的 [LLM-as-Compiler Knowledge Base](/articles/llm-as-compiler-knowledge-base) 是一個更高階的實例：一個*編譯過的、保持最新的*產出物，而非重新推導的知識。
