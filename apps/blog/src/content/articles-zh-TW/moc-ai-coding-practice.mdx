> **ai-coding-practice** 領域的內容地圖 — 23 個概念。人類與團隊如何實踐 AI 輔助的軟體工作：工作流程技術、SDLC 遙測、審查與驗證這個瓶頸，以及分工。精選入口；所有領域請見[首頁](/)。


- [加速反噬](/articles/acceleration-whiplash) — Faros 2026：AI 將人類速度主導的 SDLC 淹沒在無法吸收的產出中 — 吞吐量上升（任務 +34%、epics +66%），品質下降（錯誤 +54%、事件/PR +243%、審查時間 5 倍），差距隨採用擴大，甚至波及高成熟度的組織
- [代理式程式設計的工作組成轉移](/articles/agentic-coding-work-composition-shift) — Anthropic 的 400K 工作階段遙測，2025 年 10 月→2026 年 4 月：隨著模型進步，修復損壞程式碼的工作階段占比從 33% 降至 19%（除錯幾乎減半），而操作軟體（14%→21%）與撰寫＋資料分析（約 10%→約 20%）增加；估計任務價值上升約 25–27% — 使用方式從救火轉向端到端代理式工作
- [代理式工作系統化](/articles/agentic-work-systematization) — OpenAI Codex 研究的「系統化」邊際：從臨時使用代理（描述任務 → 代理執行 → 完成）轉向透過技能與外掛建立可重複使用的工作流程基礎設施；技能使用率從每週活躍使用者的 5.4% 上升至 26.6%（2026 年 3 月→6 月），在 OpenAI 幾乎普及（96.2%）；自訂技能（組織特定的程序脈絡）集中於具備共享慣例與團隊標準的地方 — 這是 loop-engineering 技能基元的實證對應
- [AI 作為主要作者](/articles/ai-as-primary-author) — Faros 2026：助理→作者的門檻在沒有刻意決策的情況下被跨越，AI 程式碼接受率從 20% 上升至 60% 即為標誌；「不是助理，而是作者」；人類從創作轉向監督，使問題變成寫作問題，而非審查問題
- [建造很便宜，爭論很昂貴](/articles/building-is-cheap-arguing-is-expensive) — 「在技術辯論中，程式碼會勝出」：與其白板討論，不如產生三個 PR；原型優於設計文件；減少設計文件
- [程式碼作為真相來源](/articles/code-as-source-of-truth) — 在高程式碼吞吐量下，文件會過時；將規格/技能提交至儲存庫；透過 Claude Code 進行入職；驗證規格漂移
- [運算配置器](/articles/compute-allocator) — 人類持續演變的角色：決定哪些事情值得投入運算資源；產生的 token 中約 1% 會上線，99% 是投入於對齊/溝通的鷹架；豐盛心態
- [可配置的人類參與](/articles/configurable-human-participation) — HAS-Bench（Wu et al.，arXiv 2607.04329）：一項將人類參與變成*可配置*變數的基準測試 — 人類與 LLM 代理都是共享互動圖中的一等節點，參與沿三個軸變化（五級 Human Agency Scale A1–A5、三種互動管道：澄清/回饋/控制，以及使用者角色），涵蓋六個領域的 397 項任務，同時評分結果與過程。研究發現：平等夥伴關係（A3）勝過完全自動化（A1），Pass@1 提高 +8.4、Task Score 提高 +11.5，並可挽回最多 65.4% 的自主性失敗任務；但其價值取決於配置，並非單調增加 — 更多代理權（A4）帶來遞減、且有時是負面的報酬（挽救 27 項任務／因過早或分心的介入破壞 13 項），每種問題模式都有不同的最佳管道（提交前用澄清、提交後用回饋、關鍵安全授權僅用控制 = 100% 安全），在 6/6 模式中有 5 種是最佳單一管道勝過 Full-channel，而弱代理（Llama-3.1-8B）幾乎沒有收益 — 代理必須決定*何時*詢問以及*如何*使用輸入，而不只是取得更多輸入；以上皆採用 LLM 模擬的（GPT-4.1）人類，是受控基準測試，不是現場監督負荷遙測
- [設計概念拷問](/articles/design-concept-grilling) *(樞紐)* — Matt Pocock 的 `grill-me` skill；在任何計畫之前先抵達 Brooks 的「設計概念」；反制從規格到程式碼；PRD 作為目的地文件，Kanban 作為旅程文件
- [一次性微型應用程式](/articles/disposable-micro-apps) — 每項任務建立一次性自訂 UI 來編輯計畫（「建在微型軟體之上的微型軟體」）；複製回 markdown；在豐盛心態下是合理的
- [HTML 作為新的 Markdown](/articles/html-as-the-new-markdown) — Thariq Shihipar 的論點：隨著模型進步，千行 markdown 計畫會壓垮*人類*；HTML 產物（具視覺性、互動性）讓人類持續參與其中。面向模型的 harness 縮小，而面向人類的 harness 擴大
- [活的設計系統](/articles/living-design-system) — 從儲存庫擷取出的 `design_system.html`，作為可攜、可供人類與機器讀取的真相來源；元件 playground；連接工程與非技術利害關係人
- [LLM 輔助的灰色文獻理論建構](/articles/llm-assisted-grey-literature-theory-building) — Agarwal et al. 的次要貢獻（arXiv 2607.07980）：一種可擴展的模板，從數千份實務工作者文件而非幾十次訪談建構紮根理論 — LLM 負責機械式、以引文為錨點的開放編碼（收集 38,709 份文件 → Gemini relevance judge，κ=0.75 → 使用 Thematic-LM 多代理系統，在三種刻意極化的編碼者視角下編碼 3,100 份 → 4,838 個程式碼／109,951 則引文，約 $0.35/份文件），而人類保留詮釋性的軸心/選擇性編碼；自動化後半段失敗（由下而上的處理產生 15,029 條膚淺、重複的因果陳述），因此程式碼→理論步驟仍是人工、以 LLM 作為搜尋引擎的流程 — 一項關於 LLM 在質性研究中能做與不能做什麼的分工教訓
- [外包你的思考，而非你的理解](/articles/outsource-thinking-not-understanding) — 「你可以外包思考，但不能外包理解」；理解是不可委派的人類瓶頸；知識庫是理解工具
- [規劃／執行分工](/articles/planning-execution-division-of-labor) — Anthropic 的 400K 工作階段遙測：在典型的 Claude Code 工作階段中，人類做出約 70% 的規劃決策（做什麼），而 Claude 做出約 80% 的執行決策（如何做）；每個提示會觸發約 10 個動作（使用者保留執行控制時為 8 個，Claude 控制規劃時為約 16 個） — 「人們決定建造什麼，代理決定如何建造」
- [審查作為控制點](/articles/review-as-the-control-point) — Agarwal et al.（CMU，arXiv 2607.07980）：由 3,100 份已編碼的實務工作者文件綜合而成的 26 構念/67 關係因果理論 — 審查是決定編碼代理如何影響軟體的控制點，而 AI 不會固定該影響的方向；團隊透過審查者專業、處置方式，以及如何調整審查流程來設定它（三個調節因子）。核心是審查深度＋審查者技能，由理解債務回饋迴圈貫穿；論文本身的非供應商 GitHub 遙測（250 萬+ PR）發現，代理 PR 的審查較少／合併速度快數倍／討論較少，但在可辯護的分析選擇下，趨勢會反向，而無審查率會隨時間收斂至人類基準
- [遙測與調查測量](/articles/telemetry-vs-survey-measurement) — Faros 2026：感知落後於現實，因此以調查為基礎的工程研究（DORA）會漏掉系統遙測幾乎即時捕捉到的下游 AI 損害；這也是 Faros 直接反駁 DORA「強大的基礎能保護你」結論的根據
- [AI 原生建造的三個迴圈](/articles/three-loops-of-ai-native-building) — Andrew Ng 對 0 到 1 產品提出的巢狀迴圈分類法：代理式程式設計迴圈（數分鐘，代理閉合）、開發者回饋迴圈（數十分鐘至數小時，人類閉合），以及外部回饋迴圈（數小時至數週，市場閉合）；loop engineering 一直只在最佳化最內層，而人類剩餘的工作是存在於外兩層的脈絡轉移
- [未知數作為代理式瓶頸](/articles/unknowns-as-the-agentic-bottleneck) — Thariq Shihipar 的地圖與疆域論點：你告訴代理的內容與工作實際需要之間的差距就是*未知數*；面對 Fable 級模型，人類找出這些未知數的能力 — 而非模型能力 — 決定產出品質；將 Rumsfeld 2×2 套用於提示，並加上按階段排序的引導技術目錄
- [可驗證性論點](/articles/verifiability-thesis) *(樞紐)* — LLM 會自動化你能*驗證*的事物，就如同電腦會自動化你能*指定*的事物；RL 驗證獎勵 → 鋸齒狀峰值；「可驗證 + labs 關心」；一切最終都可驗證
- [驗證成為新瓶頸](/articles/verification-as-the-new-bottleneck) *(樞紐)* — Fiona Fung：程式設計不再是瓶頸 — 驗證、審查、維護才是；左移；TDD 失去其稅負；PR 週期時間漏斗分析
- [垂直切片曳光彈](/articles/vertical-slice-tracer-bullets) — 將 Pragmatic-Programmer 的曳光彈模式套用於代理任務分解；垂直切片 > 水平分層；採用帶阻塞邊的 Kanban，而非編號階段計畫
- [Vibe Coding 與代理式工程](/articles/vibe-coding-vs-agentic-engineering) — Vibe coding 拉高下限（任何人都能建造）；代理式工程在加速的同時維持品質門檻；「>10 倍且持續擴大」；在大型專案上招募，而不是用謎題選才
