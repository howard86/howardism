## 摘要

[Andrej Karpathy](/articles/andrej-karpathy) 對三種程式設計典範的分類：**Software 1.0** 是顯式程式碼；**Software 2.0** 是學習而來的權重（你透過策劃資料集、目標函數與架構來「寫程式」）；**Software 3.0** 則是 *提示*——LLM 是一台可程式化的電腦，「上下文視窗裡的內容就是你操控直譯器的槓桿。」模型在整個網際網路上訓練，隱性地多工處理資料中的每一項任務，成為在數位資訊空間中執行運算的通用直譯器。更深層的主張是：3.0 不只是讓舊程式跑得 *更快*，它還讓整類程式變得 *不再必要*——並使從前不可能存在的資訊處理任務得以實現。

## 三種典範

| | Software 1.0 | Software 2.0 | Software 3.0 |
|---|---|---|---|
| 你寫的是 | 程式碼 | 資料集 + 目標函數 + 架構 | 提示／上下文 |
| 執行者 | CPU | 訓練好的神經網路 | 作為直譯器的 LLM |
| 「程式」是 | 原始碼 | 權重 | 上下文視窗 |

## OpenClaw 安裝程式範例

過去要安裝一個複雜的跨平台工具，意味著要寫一段為了涵蓋所有環境而「不斷膨脹、變得極其複雜」的 shell 指令稿——這是 Software 1.0 式的思維。3.0 版本則是：安裝說明就是 **一段你複製貼上給 agent 的文字**。這個 agent 自帶其智慧、檢視你的機器、執行有判斷力的動作，並在過程中迴圈除錯。「該複製貼上給你的 agent 的那段文字是什麼？這就是現在的程式設計典範。」（其推廣形式見 [Agent-Native Infrastructure](/articles/agent-native-infrastructure)。）

## MenuGen：不該存在的應用程式

Karpathy 打造了 MenuGen——拍下一張菜單、用 OCR 辨識品項、為每一道菜生成圖片——這是一款搭配影像生成基礎設施、實際運行在 Vercel 上的應用程式。後來他看到了 3.0 版本：把照片交給 Gemini 並說 *「用 Nano Banana 把這些菜餚疊到菜單上」*，模型就回傳一張在像素層面已渲染好菜餚圖片的菜單影像。「我所有的 menu gen 都是多餘的。那個 app 不應該存在。」神經網路涵蓋了整個應用程式；輸入只是圖片，輸出也只是圖片，中間沒有任何 app。

## 超越程式碼：全新的資訊處理任務

還有一個更微妙的觀點：以往的程式碼是運作於 *結構化* 資料之上。Software 3.0 讓那些從來都算不上是程式的操作得以實現。他舉的例子是 [LLM wiki](/articles/llm-as-compiler-knowledge-base)：「過去沒有任何程式碼能從一堆事實打造出一個知識庫。現在你可以拿這些文件、用另一種方式重新編譯⋯⋯把資料重新框架化後產生全新的東西。」他說這比單純的加速 *「更令人興奮」*——*重點不是我們能更快做到什麼，而是過去根本做不到的事情。*

## 延伸推論：神經網路作為宿主程序

推到極致：一台「完全神經化的電腦」——原始的影音輸入進來、由擴散模型渲染出獨屬於當下的 UI、神經網路擔任 **宿主程序**，而 CPU 則作為負責決定性輔助功能的 **協同處理器**。他把這描繪成 1950 至 60 年代那次分岔（計算機 vs. 神經網路）的反轉：傳統運算贏了第一回合、神經網路目前被虛擬化在它之上，而這層關係可能會倒轉過來。這是 [The Bitter Lesson](/articles/the-bitter-lesson) 被推到架構層面的結論。他保留地說，通往那裡的路還是「TBD」。
