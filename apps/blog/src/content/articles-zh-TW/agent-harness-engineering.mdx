## 摘要

Agent Harness Engineering 是一門設計環境、產出物與回饋迴路的學科，使 AI 編碼 agent 能夠跨越多個 context window 進行可靠且持續的工作。核心轉變在於：工程師的工作從撰寫程式碼轉移到建構讓 agent 有效運作的鷹架——明確指定意圖、結構化上下文、強制執行不變量，以及建構驗證管線。

## 細節

### 根本問題

AI 編碼 agent 在離散的 session 中工作，且 context window 有限。每個新 session 啟動時都沒有先前工作的記憶。若缺乏刻意的 harness 設計，agent 會展現出可預測的失敗模式：

1. **一次性完成** — 試圖一次建構所有東西，在實作途中耗盡上下文，留下半完成且未文件化的工作
2. **過早宣告勝利** — 看到部分進展就宣稱工作完成
3. **髒狀態** — 留下帶有 bug、未提交的變更或未文件化的進度，讓下一個 session 去解開
4. **不完整的驗證** — 將功能標記為完成卻未進行端對端測試

### 雙 Agent 架構（Anthropic）

Anthropic 為 Claude Agent SDK 提出的解決方案使用兩個專門的提示：

- **初始化 agent**（僅第一個 session）：搭建環境鷹架——撰寫 `init.sh` 腳本、建立 `claude-progress.txt` 日誌、產生結構化的 JSON 功能清單（所有需求標記為「failing」），並進行初始 git 提交。
- **編碼 agent**（每個後續 session）：讀取進度日誌和 git 歷史、執行基本冒煙測試、選擇單一功能來實作、端對端驗證（例如透過 Puppeteer MCP 用於 web 應用程式）、提交乾淨狀態，並更新進度檔案。

JSON 功能清單至關重要：agent 被指示永遠不要移除或編輯功能描述，只能在驗證後將 `passes` 從 `false` 翻轉為 `true`。選擇 JSON 而非 Markdown 是因為 agent 較不容易意外覆寫結構化的 JSON。

### 儲存庫作為記錄系統（OpenAI）

OpenAI 的 Codex 團隊建構了一個零手動撰寫程式碼的產品（約 100 萬行、約 1,500 個 PR，3–7 位工程師歷時 5 個月）。他們的關鍵架構洞見：**儲存庫本地的版本化產出物是 agent 唯一能看到的東西**——任何在 Slack、Google Docs 或人們腦中的資訊都是不可見的。

他們的方法：
- **AGENTS.md 作為目錄，而非百科全書**：一份簡短（約 100 行）的地圖，指向更深層的文件。單體式指令檔案會失敗，因為它擠壓了任務上下文、當所有東西都「重要」時就變成非指引、立即腐化，且無法進行機械式驗證。
- **漸進式揭露**：agent 從一個小而穩定的入口點開始，並被教導在哪裡深入探索。設計文件、執行計畫和技術債都在儲存庫中版本化。
- **機械式強制執行**：自訂 linter（本身由 agent 產生）強制執行架構——層之間的依賴方向、結構化日誌、命名慣例、檔案大小限制。Lint 錯誤訊息被撰寫為修復指示，注入到 agent 上下文中。
- **文件園藝**：一個定期執行的背景 agent 掃描過時的文件並開啟修復 PR。

### 大規模強制執行架構

兩個來源都收斂於一個關鍵原則：**強制執行不變量，而非實作**。定義嚴格的邊界（層依賴、邊界處的資料驗證、命名慣例），讓 agent 在這些邊界內擁有自由。

OpenAI 對每個業務領域使用嚴格的分層架構：Types → Config → Repo → Service → Runtime → UI，跨切面關注點透過單一 Providers 介面進入。這由結構測試和自訂 linter 強制執行。他們指出這種程度的架構嚴謹性通常要到數百位工程師時才會實施——有了 agent，它是早期的先決條件，因為約束能在不偏移的情況下實現速度。

### 持續熵管理

Agent 產生的程式碼庫會累積熵：agent 會複製已存在的模式，包括次優的模式。OpenAI 最初花費 20% 的工程時間手動清理「AI 垃圾」。他們的解決方案：將「黃金原則」編碼到儲存庫中，並以定期節奏執行背景 agent 任務來掃描偏差、更新品質等級，並開啟針對性的重構 PR。這如同垃圾回收——以小幅增量持續償還技術債，而非讓它複利累積。

### Harness 即服務

上述模式描述的是每個 session 的 harness。兩個 2026 年的系統展示了自然演進——**持續運行的 harness 即服務**，具有每租戶的工作區隔離：

- [Symphony](/articles/symphony)（OpenAI，2026 年 3 月）— 長時間運行的 daemon 輪詢 Linear，每個 issue 一個工作區，每張票一個 [Codex App Server](/articles/codex-app-server-protocol) session。與上述 OpenAI 來源相同的團隊；他們明確的「harness 即服務」迭代。協調器擁有工作區生命週期、重試/退避、停滯偵測和調和；儲存庫內的 `WORKFLOW.md` 是策略檔案。
- [Hermes Agent](/articles/hermes-agent)（Nous Research）— Hermes Gateway 以 systemd 或 launchd 運行；每使用者 session 隔離；允許清單 + DM 配對授權；cron 任務傳送到指定的主頻道。

兩者的收斂設計選擇：

1. **Daemon 優先部署** — 長時間運行的服務，而非每次呼叫的 CLI。
2. **每租戶工作區隔離** — 每個 issue（Symphony）或每個使用者（Hermes）。
3. **容器後端作為信任邊界**（Docker、Singularity、Modal、Daytona）而非逐命令的核准提示。Hermes 在容器後端下明確停用危險命令檢查，原則是「容器就是安全邊界」。
4. **儲存庫版本化的 markdown 作為控制平面** — Symphony 的 `WORKFLOW.md`、Hermes 的 `AGENTS.md`/`SOUL.md`，與 session 層級的 `CLAUDE.md`/`AGENTS.md` 目錄式紀律相同的模式。
5. **預設無持久化協調器資料庫** — Symphony 明確選擇追蹤器 + 檔案系統進行重啟恢復；Hermes Gateway 狀態僅為檔案系統。

Symphony 的演進強化了上述原則。他們的第一版將 agent 視為嚴格的狀態機節點——Codex *僅*被要求實作票中的任務。他們發現一旦模型能力成長到足以「建立多個 PR 以及閱讀審查回饋並處理它」，這就太過限制，於是轉向給予 agent **目標 + 工具，而非狀態轉換**。這是「強制執行不變量，而非實作」應用於協調層（見 [Ticket-Driven Agent Orchestration](/articles/ticket-driven-agent-orchestration)）。

對於協調器與編碼 agent 之間的整合邊界，Symphony 使用 [Codex App Server protocol](/articles/codex-app-server-protocol)——透過 stdio 的 JSON-RPC，具有延續回合和動態工具呼叫——使合約明確且版本容忍。該協議的**動態工具呼叫**功能也是一個值得注意的 harness 原語：協調器實作的工具可以包裝子 agent 不應看到的憑證（例如 Symphony 的 `linear_graphql` 工具代理已認證的 GraphQL，而不將 Linear 存取權杖給予子 agent 容器）。

### 人類的角色

在兩個系統中，人類在不同的抽象層工作：
- 排定工作優先順序，並將使用者回饋轉化為驗收標準
- 設計環境和回饋迴路
- 驗證結果並提供品味/判斷
- 當 agent 遇到困難時，診斷缺少什麼能力（工具、護欄、文件）並將其回饋到系統中——始終透過 agent，而非直接撰寫程式碼
