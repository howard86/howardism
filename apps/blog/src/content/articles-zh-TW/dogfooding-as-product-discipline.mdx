## 摘要

跨來源共同指向的主張是：**產品直覺是建立出來的，不是與生俱來的；而建立它的方法，就是不懈地第一手使用你所交付的產品。** [Fiona Fung](/articles/fiona-fung) 稱之為「螞蟻糧」（Anthropic 對 dogfooding 的稱呼），並指出這正是工程師不斷向她追問的答案：要如何鍛鍊產品直覺這塊肌肉？持續使用產品，直到你「從骨子裡感受到它」。同樣的紀律也反覆出現在 [Cat Wu](/articles/cat-wu) 的午餐時段 vibe-check（[Claude Character as Product](/articles/claude-character-as-product)），以及 [John Glasgow](/articles/john-glasgow) 的「我和每一位客戶都在同一個 Slack 頻道裡」（[Founder-Led Sales Discipline](/articles/founder-led-sales-discipline)）。當 agents 讓生成變得廉價，*品味* 就成了稀缺投入，而 dogfooding 就是訓練它的課表。

## 為什麼產品直覺會成為稀缺技能

當寫程式變得廉價（[Verification as the New Bottleneck](/articles/verification-as-the-new-bottleneck)），限制因素便轉移到*知道什麼值得打造，以及做出來是否夠好*——也就是產品品味。Fung 現在招聘的兩種工程師特質，是「具備產品直覺的創意建造者」與「深厚的系統專業」。前者就是由 dogfooding 餵養出來的。這是 [Engineer PM Convergence](/articles/engineer-pm-convergence) 與 [Evals as Product Spec](/articles/evals-as-product-spec) 在組織層級上的印證：當 harness 不斷縮小，品味與判斷力就是持久的人類技能。

## Fung 的方法：從骨子裡感受到它

她建立產品直覺的方法如下：

1. **Dogfood**——尤其是管理者；在 Claude 之前，他們往往根本沒有時間使用產品。「你真的會開始從骨子裡感受到自己的產品；你會記得自己當時試圖解決的是什麼問題。」沒有這件事，「你做產品決策時就會根據指標、儀表板或 PowerPoints。」
2. **交付並反覆迭代。**
3. **與客戶交談**——她目前的熱情專案，是協助經營小型企業的朋友（餐飲業者）上手 [Cowork](/articles/cowork)，並觀察他們在哪裡卡住：「看到我們在 onboarding 上還有這麼多可以做得更好的地方，真的非常令人謙卑。」（這也連結到 Anthropic 的 Claude for Small Business。）

**Mr. Peanut 的故事**是校準品味的軼聞：她在 CLI 裡寫了一個節日版的「雪人」Claude；她的設計師卻說那看起來完全不像雪人——「你把 Claude 變成了 Mr. Peanut。」人類的產品／設計品味捕捉到了建造者自己看不見的東西，正是[模型所缺乏的那種判斷](/articles/jagged-intelligence)。

## Claude 作為跨職能缺口填補者（另一面）

Dogfooding 會揭露你的弱點；接著 Claude 便能在*所有*職能中填補這些缺口：使用 Claude Code 的設計師可以透過 Claude 自己修正細節與 UX，而不必先畫紅線再交給工程團隊（從而閉合迭代迴圈）；自稱冗長型工程師的 Fung，則把 Claude 當作「內容設計夥伴」來處理短文案。「Claude 正在強化你可能沒那麼擅長的所有領域。」Dogfooding 告訴你缺口在哪裡；工具幫你補上缺口——但*察覺缺口的品味*仍然是人類的事。

## Codex 的轉折：降低自己的流程品質，來改進產品（Ambrosino）

[Andrew Ambrosino](/articles/andrew-ambrosino)（[OpenAI](/articles/openai) Codex）提供了最極端形式的 dogfooding 約定。Codex 桌面應用程式「完全由 dogfooding 迴圈提供資訊」——但其中有一個刻意令人不舒服的倒置：

> 「我們所有人都希望盡可能在*應用程式裡完成所有事，即使它不是最佳工具*，這樣它才有機會成為最佳工具。我們常常**不改善自己的流程，好讓我們能把產品做得更好**，這是一個令人極度不舒服的處境。」

這顛覆了一般的 dogfooding。通常的交換條件是「使用你的產品來*找出*缺口」。Ambrosino 的做法則是「*拒絕更好的外部工具*並承受痛苦，讓缺口被感受得足夠強烈，迫使產品把它補上」。團隊自己的工作流程被刻意維持在更糟的狀態，讓產品承受變好的壓力——這是一種自我施加的[限制](/articles/verification-as-the-new-bottleneck)，讓品味與路線圖持續和親身經歷的摩擦緊密相連。

這起初是一個**個人** dogfooding 迴圈：Ambrosino 對原始應用程式的目標，是「讓它成為我寫程式所使用的東西」。這個良性循環——「我做不到這件事 → 我修好它，讓自己做得到 → 現在我能做更多事」——一開始只在他身上運作；後來，隨著應用程式成熟，他自己的使用方式從寫程式轉向產品探索／引導，因此他又把使用方向對準*那個*（「現在我需要建立一個替這件事建模的試算表，對這個領域的所有工作進行內部深度研究」）。讓自己的使用方式對準產品接下來必須解決的問題，就是這項紀律。

| 來源 | Dogfooding 的做法 |
|---|---|
| [Fiona Fung](/articles/fiona-fung) | 「螞蟻糧」——管理者透過 dogfood 維持產品直覺；讓小型企業上手，親身感受 onboarding 的痛點 |
| [Cat Wu](/articles/cat-wu) | 午餐時段對模型進行 vibe-check，作為 eval 紀律（[Claude Character as Product](/articles/claude-character-as-product)） |
| [John Glasgow](/articles/john-glasgow) | 創辦人加入每個客戶的 Slack 頻道；參與銷售通話來推動工程（[Founder-Led Sales Discipline](/articles/founder-led-sales-discipline)） |
| [Dan Carey](/articles/dan-carey) | 「使用 [Claude Design](/articles/claude-design) 來設計 Claude Design」——每天打造並 dogfood 自己的工具；Claude 會先分析*每一段*使用者對話 |
| [Andrew Ambrosino](/articles/andrew-ambrosino) | *即使 Codex 應用程式是錯誤的工具，也要使用它*，迫使它成為正確的工具；讓自己的流程維持更糟，來讓產品變得更好 |

五者都把*直接、反覆、第一手接觸實際運作中的產品／客戶*視為品味不可妥協的來源——而 Ambrosino 進一步指出，你可能必須*把自己的流程扣押為人質*。

## 開放問題

- 當團隊*就是*使用者（Claude Code），或與使用者非常接近（Cat Wu、Boris）時，Dogfooding 很有效。對於與自己非常不同的使用者，要如何建立產品直覺——正如 Glasgow／Fung 的小型企業工作所暗示的，「與客戶交談」是否能完全取代它？
- Dogfooding 能否擴展，還是它其實暗中限制了 AI 原生產品組織維持品味驅動的規模，直到組織再次退回依賴儀表板？
