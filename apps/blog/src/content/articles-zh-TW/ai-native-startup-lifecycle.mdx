## 摘要

[Anthropic](/articles/anthropic) 2026 年將經典的 Lean/YC 新創路徑（驗證 → 募資 → 招人 → 開發 → 再募資 → 成長 → 再招人）重新定義為四個階段，明確假設 AI 為核心基礎設施：**Idea → MVP → Launch → Scale**。結構性的改變在於*每個新階段不再需要更大的團隊、不同的技能組合、或新一輪募資*。「精實 10 人獨角獸」被定位為刻意追求的目標，而非刻苦求生的異數。每個階段保留傳統的退出標準（問題-解決方案契合 → 產品市場契合 → 可重複成長 → 可防禦規模），但*通過的路徑*從數季壓縮為數週。

## 四個階段

### Idea — 以研究為導向的驗證

**目標：** 在投入開發資源之前，收集質性證據證明真實問題存在，且提出的解決方案能解決該問題。

**退出標準（三項皆須為是）：**
1. 問題是真實且具體的 — 你能指出誰有這個問題、多常發生、多嚴重、他們目前怎麼處理。
2. 你的解決方案解決的是*實際*問題（不是你最初假設的那個；驗證過程經常重塑問題）。
3. 足夠的訊號證明值得開發 — 質性證據顯示投入 MVP 是理性決策而非信仰之舉。

**階段風險**（[Problem-Solution Fit Discipline](/articles/problem-solution-fit-discipline)）：
- 把開發誤認為驗證（一個能運作的原型*不是*問題真實存在的證據）。
- 過早擴張（agentic coding 能讓執行遠超已驗證的方向）。
- 喪失客觀性（AI 順從你的方向 — 確認偏誤獲得了研究引擎）。

**AI 的角色：** 研究夥伴。假說的魔鬼代言人。以壓力測試過的假設建構 TAM/SAM/SOM。按層級（直接、間接、潛在收購者、相鄰）繪製競爭者地圖。訪談框架審計以揭露引導性或前瞻性問題。Claude Code 僅在最末端進入，用於建構輕量原型*作為客戶對話中的道具* — 而非產品本身。

### MVP — 將已驗證的問題轉化為可運作的產品

**目標：** 最小、最聚焦的迭代，將解決方案放到真實使用者面前並產生產品市場契合的證據。同等重要的次要目標：開發時不累積會複利增長的 [Agentic Technical Debt](/articles/agentic-technical-debt)。

**退出標準：** 特定可辨識的使用者回來使用（留存）、付費（營收）、或告訴他人（推薦）。實用的石蕊測試：
- **Sean Ellis 測試** — >40% 的活躍使用者表示如果無法再使用會「非常失望」。
- **努力測試** — 留存開始拉動而非推動；維持使用者參與所需的英雄式創辦人能量下降。

**階段風險：**
- [Agentic Technical Debt](/articles/agentic-technical-debt) — 以複利方式累積，而非線性累加。
- 虛假的產品市場契合 — 來自創辦人朋友、投資人投資組合、HN 爆量的早期牽引力無法預測第 12 週的行為。
- [Zero-Friction Scope Creep](/articles/zero-friction-scope-creep) — 當功能只需一個下午就能完成，基於成本的強制約束就消失了。
- 因經驗不足而不安全 — agentic coding 產出的是功能性程式碼，不是天生安全的程式碼。在任何使用者接觸應用程式之前進行安全審查是最低限度的負責門檻。

**AI 的角色：** Claude Code 作為主要開發工具，但僅在架構和範圍以 CLAUDE.md 上下文文件定義*之後*。Claude 在上線*之前*設計衡量框架。[Cowork](/articles/cowork) 運行營運層（使用者聯絡清單、外展序列、回饋綜合）。

### Launch — 將牽引力轉化為可持續的成長引擎

**目標：** 可重複的、通路驅動的成長 + 生產環境強化的基礎設施 + 釋放創辦人注意力的營運系統。從親自做事到設計做事的系統的轉變。

**退出標準（三要素）：**
1. 成長是可重複且通路驅動的（CAC、LTV、回收期已知且可防禦）。
2. 產品能處理生產工作負載（安全/合規就緒；可靠性在真實條件下維持）。
3. 營運不再以創辦人為瓶頸（創辦人不再親自處理支援、分類、衝刺規劃或報告）。

**階段風險：**
- 技術債到期 — MVP 階段的捷徑現在開始計息。
- 創辦人成為瓶頸 — 應該花一小時的決策花了一週；支援請求堆積因為只有創辦人知道答案。
- 安全與合規不再可延後 — 處理客戶資料、支付或受監管產業會翻轉風險態勢。
- 準備好之前就擴張 — 新市場引入新變數，瓦解你解讀自身數據的能力。

**AI 的角色：** 三個 Claude 介面全面使用，複利累積。Claude Code 審計 MVP 程式碼庫的結構性弱點。Claude 分類並排序修復工作。Cowork 審計創辦人的營運負載並分類（自動化 / 委派 / 僅創辦人）。

### Scale — 建立可防禦的事業

**目標：** 系統性的、組織支撐的成長；透過累積深度建立可防禦的護城河（見 [Compounding Data Moat](/articles/compounding-data-moat)）。創辦人角色從建造者重新定位為對外高管（分析師簡報、IPO 路演）。

**退出標準：** 門檻事件，而非單一里程碑。三種典型形式 — 不再需要外部資本的可持續獲利、IPO 就緒、或被收購。三者皆需要系統性可審計的成長 + 經得起檢視的產品護城河 + 營運成熟的組織。

**定義性問題：**「如果一個資金充裕的既有業者今天複製你的產品，你的使用者會留下嗎？」

**階段風險：**
- 委派營運層（信任 AI 系統的心理 + 結構性困難）。
- 擴展技術營運（客戶要的是基礎設施夥伴級的可靠性，不只是產品功能）。
- 擴展組織功能（招聘、薪資、會計、法務 — 無論人數多少）。
- 建立真正的 GTM 功能（有機的創辦人主導成長會碰到天花板；需要行銷、銷售、分析師關係）。

**AI 的角色：** 由小團隊運行的企業級組織的營運層。Claude Code 將程式碼強化至企業標準（日誌、監控、事件回應、可執行 SLA 的可觀測性）。Cowork 運行企業支援營運（工單路由、升級、續約追蹤）。Claude 從零建構 GTM 資源（市場區隔、訊息傳遞、銷售劇本、分析師關係策略）。

## 相較於經典生命週期的結構性新意

| 傳統生命週期假設 | AI-native 版本 |
|---|---|
| 每個階段需要更大的團隊 | 人數可以從頭到 Scale 保持不變 |
| 每個階段需要新一輪募資 | 資本效率使 Series A 前獲利成為可能 |
| 每個階段需要不同的技能組合 | 創辦人 + Claude 介面取代大多數功能性招聘 |
| 驗證受限於「你能不能做出來」 | 驗證受限於*紀律*（開發門檻已降低） |
| 範圍蔓延受限於工程成本 | 範圍蔓延受限於*書面範圍紀律*（成本門檻已降低） |
| 技術債線性累積 | 沒有持久上下文檔案時會[複利累積](/articles/agentic-technical-debt) |
| GTM 動作需要團隊才能擴展 | Cowork 可運行企業級營運層 |

## 隱含論點

這本手冊讀起來像是 Anthropic 的主張：*創辦人的工作沒有改變* — 找到真實問題、建造解決它的東西、擴展它 — 但*每個階段的瓶頸已經轉移*。瓶頸不再是「你能不能做出來」而是「你知不知道該做什麼、開發時能否保持紀律、以及你是否編碼了足夠的領域深度使競爭者無法複製」。見 [Printing Press Software Democratization](/articles/printing-press-software-democratization) 的宏觀類比和 [Founder as Agent Orchestrator](/articles/founder-as-agent-orchestrator) 的角色意涵。

## 與其他 wiki 來源的張力

- **vs. [AI Employee Framing](/articles/ai-employee-framing)（HBR Kropp 等人，2026 年 5 月）：** 手冊大量使用「編排 agents」/「AI 作為隨叫隨到的專家」/「AI 作為工程團隊」/「AI 作為營運團隊」的框架。HBR 的實證研究顯示，正是這種擬人化框架可測量地降低個人責任感（−9pp）、增加不必要的升級（+44%）、並減少錯誤捕捉（−18%）。手冊未回應此證據。應用手冊的有紀律創辦人應在內部保持*工具*框架的責任感，即使使用「永遠可用、永不阻塞的工程師」心智捷徑。
- **vs. [Harness Shrinkage as Models Improve](/articles/harness-shrinkage-as-models-improve)：** 手冊將 Claude 介面視為固定基礎設施（Chat / Cowork / Code）。但 Anthropic 自己的論點（Boris Cherny、Cat Wu）是 harness 本身每次發布都會縮小。圍繞 2026 年 harness 能力建構永久工作流程的創辦人應預期這些工作流程隨著能力向內遷移而需要重寫。
- **vs. [Claude Code Best Practices](/articles/claude-code-best-practices)：** 手冊建議每次 Claude Code session 以範圍 + CLAUDE.md 上下文開始，以日誌條目結束。這比官方最佳實踐文件更嚴格的紀律，被框架為創辦人對抗 [Agentic Technical Debt](/articles/agentic-technical-debt) 的特定防護。

## 開放問題

- 手冊未提供人力/資本壓縮主張的量化證據（無中位數達到 PMF 的時間、無達到 PMF 時的人數、無失敗率數據）。「精實 10 人獨角獸」被斷言為刻意目標，文件本身無案例研究證據。
- 資源區段中的創辦人故事（Carta Healthcare、Anything、Cogent、Airtree、Duvo、Zingage、Kindora、Wordsmith）是簡短引述 — 無一有已發表的成果或可比較基線數據。
- CB Insights 42%「做了沒人要的東西」的數據來自 AI 之前的時代；手冊預測此比率會攀升但未引用 2026 年的測量。
- 與 HBR 責任感研究結果（如上）的張力未解決。手冊的編排框架讀起來正是 HBR 實驗條件所*反對*測試的框架。
