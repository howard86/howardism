## 摘要

[Thariq Shihipar](/articles/thariq-shihipar) 的第三個 HTML 優先工作流程：維護一份 **`design_system.html`** 檔案——一個人機皆可讀的產出物——作為應用程式視覺語言的可攜式單一事實來源。不同於靜態的風格指南文件，它會*實際渲染*：色彩調色盤、字型比例尺、間距規則，以及核心元件（按鈕、輸入框、卡片）的互動範例全部即時呈現在頁面上。每次開始新工作時，你只需將這一個檔案交給 Claude 作為上下文，它產出的一切都會符合品牌規範。

這是 [HTML as the New Markdown](/articles/html-as-the-new-markdown) 理念應用於設計系統一致性這個長期難題：同一份人類可以目視檢查的產出物，也是模型用來保持一致性的產出物。

## 如何建構

1. **萃取設計 DNA。** 將 Claude 指向一個專案資料夾——或同時指向多個 GitHub 儲存庫（例如行銷網站*和*應用程式）——並要求它萃取設計系統。它會分析 CSS、元件和樣式來推斷視覺語言。
2. **產生 `design_system.html` 產出物。** Claude 將萃取出的系統渲染為單一 HTML 檔案，以視覺方式呈現調色盤、字型比例尺、間距，以及互動式元件範例——不是元件的描述，而是元件本身。
3. **探索變體並分享。** 透過 Claude Design 等工具，建構帶有互動旋鈕和滑桿的元件遊樂場（「這個按鈕加大 padding 會長什麼樣？換個邊框呢？」），即時探索各種變體。

## 為什麼稱為「活的」

傳統的風格指南會漂移：文件說一套，程式碼做另一套。從程式碼中*萃取*並*渲染*為真實元件的 `design_system.html` 更難造假——你看到的就是元件實際的行為。而且因為它是單一可攜檔案，可以隨處帶著走：交給新專案、新功能或隊友，它就成為維持產出一致性的參考依據。它是一次**持久的**[運算](/articles/compute-allocator)配置——產生一次，作為上下文無限期重複使用——與用完即丟的[微型應用](/articles/disposable-micro-apps)形成對比。

## 串接工程團隊與公司其他部門

這份產出物最大的回報在於組織層面。一個渲染好的元件頁面讓非技術利害關係人——行銷人員、PMMs、任何需要製作簡報或影片的人——只需前往一個頁面，就能看到每個元件在各種狀態下的樣貌，並取得高保真模型，「不用去煩工程師。」[Claire Vo](/articles/claire-vo) 就是用自己的元件視覺化頁面做到這件事。因此，設計系統不僅串接了人類↔AI，更串接了**工程團隊↔公司其他部門**——與 [HTML as the New Markdown](/articles/html-as-the-new-markdown) 對計畫所宣稱的人類可讀性回報如出一轍。
