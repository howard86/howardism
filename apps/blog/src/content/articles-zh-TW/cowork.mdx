## 摘要

[Anthropic](/articles/anthropic) 的知識工作 agent 產品，是 [Claude Code](/articles/claude-code) 的姊妹產品。Claude Code 針對的是輸出為*程式碼*的工作，而 Cowork 針對的是輸出為*非程式碼*的工作：簡報投影片、收件匣分類、客戶摘要報告、上線文件、會議準備。產品由 [Cat Wu](/articles/cat-wu) 共同主導。

## 定位（Cat Wu 的框架）

- **[Claude Code](/articles/claude-code) (CLI)** — 一次性程式碼任務；CLI 最先獲得新功能；「所有工具中最強大的」
- **Claude desktop / Claude Code in IDE** — 前端工作，需要預覽面板的場景；更適合不習慣終端機的非技術使用者
- **Claude Code mobile / web** — AFK 時啟動任務；在推出之前是「缺失的產品」——人們會在戶外用手機分享筆電網路
- **Cowork** — 任何輸出不是程式碼的工作

## Cat Wu 如何使用它

連接 Google Calendar、Slack、Gmail、Google Drive——「給越多上下文越好。」範例工作流程：晚上將草稿大綱加上相關 Twitter / 上線聊天室連結餵給 Cowork，隔天早上回來就有一份 20 頁精美簡報，再經過一輪回饋迭代即可完成。Anthropic 的設計系統已載入，因此簡報看起來就像設計師做的。

## 人們為什麼使用它（根據 Cat Wu）

「Applied AI」團隊——內部第二大 token 消耗者（僅次於工程團隊）——使用 Cowork 進行：
- 客戶會議準備（「摘要報告」工作流程：昨天的問題、行動項目、預計完成時間研究）
- 收件匣清零 / Slack 清零分類
- 上線計畫、內部文件、從 Salesforce + Gong + Slack 提取的客製化客戶簡報

## 它如何融入更大的 Anthropic 路線圖

- [Boris Cherny](/articles/boris-cherny) 將 Cowork 定位為通用 [MCP and computer use](/articles/mcp-and-computer-use) 整合的場域——與 Claude AI 使用相同的 MCP 連接器，加上 Anthropic 在**computer use** 方面的領先地位，用於缺乏 MCP 的軟體（在 Opus 4.7 上「相當不錯」）。
- 「我們公司已經沒有任何手動撰寫的程式碼了」——Cowork 涵蓋了這個宣稱的非程式碼那一半（SQL、簡報、營運文件全部由模型生成）。

## 在創辦人手冊中的角色（2026 年 5 月）

[The Founder's Playbook](/articles/ai-native-startup-lifecycle) 將 Cowork 定位為 AI 原生新創公司在每個階段的**營運層**：

- **構想階段** — 自動化客戶外展與排程。Cowork 使用經驗證的目標輪廓來研究潛在客戶、撰寫個人化外展訊息、透過 MCP 連接 Gmail/Calendar、管理對話串、執行第 7 天追蹤。
- **MVP 階段** — 回饋循環的後勤工作：向早期使用者外展、安排回饋會議、結構化的錯誤與功能收集、每週綜合報告。
- **上線階段** — 在營運層取代創辦人的注意力：審計創辦人的重複性任務、分類為自動化/委派/僅創辦人處理、為自動化候選項建立工作流程邏輯。PM 營運系統：衝刺排程、錯誤分類路由、每週指標彙整。
- **規模化階段** — 企業級營運：工單路由、升級工作流程、由產品變更觸發的文件更新、續約追蹤、報告節奏。加上 GTM 戰術執行：內容管線、外展序列、分析師簡報後勤、CRM 維護。

手冊的框架：Cowork + [Claude Code](/articles/claude-code) 共同賦予「小團隊大組織的支援態勢」。Skills 被強調為將重複性創辦人工作流程編碼化的表面。

## 關鍵引述

- （Cat Wu）「如果我在做的東西輸出是程式碼，我會用 Claude Code 或 desktop 或 Claude Code on mobile。如果輸出是任何非程式碼的東西，我就用 Cowork。」
- （Boris Cherny）「Claude design 我覺得是一個很好的例子。它今天已經相當不錯了。它會變得好很多。」（指的是他願景回答中 Cowork 類產品。）

## 開放問題

- Cowork 的 harness 與 [Claude Code](/articles/claude-code) 的相比如何？兩者都提供 skills、MCP、sub-agents——但非程式碼輸出的失敗模式不同（沒有測試套件、沒有編譯器、沒有 diff 可審查）。
- Cowork 類輸出的 eval 紀律是什麼？Cat Wu 說記憶功能從 evals 中受益很大；但簡報品質如何衡量仍不清楚。
