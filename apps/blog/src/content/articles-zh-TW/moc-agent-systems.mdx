> **agent-systems** 領域的內容地圖——23 個概念。Harness 工程、代理迴圈與編排、上下文管理、協定與工具基礎設施（MCP、應用程式伺服器），以及子代理。精選入口；所有領域請見 [Home](/)。


- [代理上下文檔案](/articles/agent-context-files) — 跨廠商的 markdown 作為控制平面模式：由儲存庫版本控管的純文字檔（CLAUDE.md / AGENTS.md / SOUL.md / WORKFLOW.md / SPEC.md /.cursorrules），在 project / personality / workflow / spec 層級依角色拆分，以設定代理行為
- [代理 Harness 工程](/articles/agent-harness-engineering) — 為長時間執行的 LLM 代理搭建腳手架的模式：環境設計、逐步揭露上下文、機械式架構強制、代理程式碼審查
- [代理迴圈模式](/articles/agent-loop-pattern) — `/loop`（由 cron 排程）與 Ralph Wiggum（消化待辦）的迴圈，作為次世代代理基元；AFK 執行、平行扇出、「迴圈就是未來」
- [代理原生基礎設施](/articles/agent-native-infrastructure) — 世界目前仍為人類打造，必須改寫成適合代理；「我要複製貼上什麼給我的代理？」；感測器／致動器；代理對代理的表徵
- [代理品質飛輪](/articles/agent-quality-flywheel) — 將 Google 的 eval-fix 迴圈包裝成由你的程式碼代理驅動的 skill：建置與測試 → 發布與監控 → 學習與精進，擴充為五個階段（準備資料／執行推論／評分／分析失敗／最佳化）；輸入是白話的擔憂，輸出是指標選擇與前後差異；Synthetic User Simulator 負責啟動，production OTel traces 使結果更精準
- [為下一個模型打造](/articles/build-for-the-next-model) — 原型化那個幾乎能運作的東西，而不是已經能運作的東西：押注下一個具體的模型版本（而非遙遠未來的 AGI）會修正工程無法解決的問題；Claude Design 的 Opus 4.7 回報，以及 OpenAI 的「二月的 Codex app 到了十一月就會失敗」是最清楚的案例——相同的產品形狀、不同智慧程度的版本、不同的結果
- [Claude Code 自動模式](/articles/claude-code-auto-mode) — Claude Code 的權限模式，使用分類器自動核准安全的工具呼叫並封鎖高風險呼叫；介於預設模式與 `--dangerously-skip-permissions` 之間的折衷方案
- [Claude Code 最佳實務](/articles/claude-code-best-practices) *(入口)* — Anthropic 對有效使用 Claude Code 的指南：上下文管理、由驗證驅動的開發、探索→規劃→編碼工作流程、環境設定
- [用戶端代理最佳化](/articles/client-side-agent-optimization) — AgentOpt 將由開發者控制的代理最佳化（每個角色使用不同模型、預算、路由）界定為不同於伺服器端 serving 的概念；組合抽象化；最佳與最差組合之間存在 13–32 倍成本差距
- [Codex App Server Protocol](/articles/codex-app-server-protocol) — 無頭 Codex 工作階段使用的 JSON-RPC stdio 協定：initialize/initialized/thread-start/turn-start 交握、延續回合重用 thread_id、用於注入 token 隔離工具的動態工具呼叫
- [上下文視窗智慧區](/articles/context-window-smart-zone) *(入口)* — 智慧區與愚蠢區（Dex Hardy / Matt Pocock）：注意力的二次方縮放、獨立於宣稱上下文大小的約 100K 標記分界；清除並重新開始 > 壓縮；狀態列 token 計數是不可或缺的紀律
- [代理的深層模組](/articles/deep-modules-for-agents) — 將 Ousterhout 的深層／淺層模組應用於適合代理的程式碼庫；推送式／拉取式指令傳遞；在新上下文中進行審查；Sandcastle 三代理模式
- [深度研究代理](/articles/deep-research-agents) — 將複雜查詢拆解、反覆搜尋多元來源，並綜合成結構化且附引用報告的代理系統——不同於單次 QA；DRACO 顯示，編排（Perplexity）勝過搭配工具的裸模型，而事實準確度是薄弱軸線
- [看似成功的失敗](/articles/failures-that-look-like-success) — 一種安靜的代理失敗類型：一切讀起來都正常——自信的答案、合理的計畫，甚至正確的內部狀態——但面向使用者的結果是錯的；Google 的飛輪示範捕捉到代理在正確執行 memorize 呼叫後仍回傳過時值，並悄悄跳過自我回報指令；必須透過 trace 層級的評分規則偵測，而不是略讀輸出
- [模型改進下的 Harness 縮減](/articles/harness-shrinkage-as-models-improve) *(入口)* — 每次模型版本更新，提示腳手架都會縮減；Cat Wu 的刪減紀律；Boris Cherny 主張「一年後只剩 100 行程式碼」；機械式驗證仍是承重結構
- [潛在空間與確定性空間](/articles/latent-vs-deterministic-space) — Garry Tan 用來診斷代理系統錯誤的框架：計算存在於兩個地方——潛在空間（LLM：品味、判斷、模糊意圖詮釋，由 markdown 引導）與確定性空間（產生的程式碼、外部狀態）——而多數 AI 工程失敗，都是計算發生在錯誤的一側
- [LLM 作為編譯器的知識庫](/articles/llm-as-compiler-knowledge-base) — Karpathy 的架構：LLM 將原始文件逐步編譯成持久且互相連結的 wiki，以 4 階段 ingest→compile→query→lint pipeline 取代 RAG
- [迴圈工程](/articles/loop-engineering) — 不再親自充當代理的提示者，而是設計負責提示它的系統：由五個產品原生基元（automations、worktrees、skills、connectors、sub-agents）加上外部記憶組成的遞迴目標迴圈；跨 Codex 與 Claude Code、與工具無關；槓桿點從提示撰寫轉移到迴圈設計
- [MCP 與電腦使用](/articles/mcp-and-computer-use) — Anthropic 的兩種互補連接機制：MCP 用於結構化的程式化存取（Salesforce/Drive/Gmail/Slack/Figma 與利基產業系統）；當不存在 MCP 時，computer use 作為驅動 GUI 的萬用方案；Boris Cherny 說「對模型而言，它就只是 token」
- [最佳化器與評估器解耦](/articles/optimizer-evaluator-decoupling) — eval-fix 迴圈中的架構規則：提出修正方案者（程式碼代理、自動最佳化器、人類）永遠不負責評分；由獨立的評估服務為結果評分，因為會評分自己工作的最佳化器會學會操弄指標，而不是改進代理
- [平行代理編排](/articles/parallel-agent-orchestration) — OpenAI Codex 研究中的並行與執行時間數據：密集型使用者工作流程中，人類監督代理團隊而非直接完成工作——28.6% 的 OpenAI 使用者在一週內曾達到 5 個以上並行代理（相較之外部使用者約 64–67% 零並行執行），而 p99 OpenAI 使用者每天執行約 71 個代理小時（自 2026 年 4 月增加 88%）；執行緒互動模型讓一個人能委派、監控並審查許多同時進行的串流——這是「創辦人即代理編排者」背後第一批硬數據
- [儲存庫探索子代理](/articles/repository-exploration-subagent) — FastContext 主張，儲存庫探索（讀取／搜尋／定位）應與解題解耦，交給專責的唯讀子代理；該子代理平行發出工具呼叫，回傳精簡的檔案行號引用，保持解題者的上下文乾淨——可將主要代理 token 降低最多 60%，並將 SWE-bench 解題率提升至 5.5%
- [票券驅動的代理編排](/articles/ticket-driven-agent-orchestration) — 讓 Symphony 運作的反轉：以票券作為工作單位（而非工作階段／PR）、DAG 依賴、可由代理擴充的工作圖，以及「目標，而非轉移」
